# Руководство для AI-агентов (Cursor, Codex и др.)

Документ для агентов, работающих с репозиторием **singbox-launch**. Следуй ему при реализации задач, рефакторинге и ответах на вопросы по коду.

---

## 0. Scope агента

**Агент может:**
- читать любые файлы репозитория;
- изменять только файлы, необходимые для выполнения **TASKS.md** текущей фичи;
- создавать новые файлы только если это явно указано в **SPEC.md** или **PLAN.md**.

**Агент не должен:**
- изменять архитектуру без явного указания в SPEC.md;
- перемещать файлы;
- менять public API (экспортируемые типы, сигнатуры) без необходимости для задачи.

---

**Архитектура и производительность:** соблюдать **архитектурные инварианты** и **ограничения по производительности** из `todo/constitution.md` (разделы 2.5, 5, 10).

---

## 1. Что читать в первую очередь

| Документ | Назначение |
|----------|------------|
| **todo/constitution.md** | Неизменяемые принципы: приоритеты, архитектура, запреты, приватность, стиль |
| **todo/IMPLEMENTATION_PROMPT.md** | Философия разработки, требования к коду, Definition of Done, ограничения (Git, консоль) |
| **docs/ARCHITECTURE.md** | Структура проекта, зоны ответственности, потоки данных, точки входа |
| **todo/README.md** | Формат задач (Spec Kit) и workflow |

Перед реализацией фичи из `todo/{FEATURE}/` обязательно прочитай **SPEC.md**, **PLAN.md**, **TASKS.md** этой фичи и применяй **IMPLEMENTATION_PROMPT.md**.

---

## 2. Формат задач (Spec Kit)

Крупные фичи описаны в `todo/{FEATURE_NAME}/`:

| Файл | Содержит |
|------|----------|
| **SPEC.md** | Что и зачем — проблема, требования, критерии приёмки, структуры данных |
| **PLAN.md** | Как строить — компоненты, изменения по файлам |
| **TASKS.md** | Чеклист задач по этапам |
| **IMPLEMENTATION_REPORT.md** | Заполняется после реализации — статус, изменения, дата |

**Workflow:** SPEC → PLAN → TASKS → реализация по TASKS с учётом IMPLEMENTATION_PROMPT → IMPLEMENTATION_REPORT.

Если пользователь просит «реализовать задачу» или «сделать по ТЗ» — уточни или возьми задачу из соответствующего **TASKS.md** и не выходи за рамки **SPEC** и **PLAN**.

**Неоднозначность SPEC/PLAN:** если SPEC.md или PLAN.md не определяют поведение однозначно (архитектура, UX, граничный случай) — агент **обязан остановиться** и запросить уточнение у пользователя, по возможности сразу предложив вариант. Не делать самостоятельно предположения, меняющие архитектуру или UX.

---

## 3. Язык и стиль

- **Отчёты, документы и общение** — преимущественно **русский** (ответы пользователю, IMPLEMENTATION_REPORT, пояснения в чате, документация в репозитории).
- **Комментарии и docstring** в коде — на **английском**; длинные пояснения допустимы на русском (см. constitution 8.3).
- **Имена** в коде (переменные, функции, пакеты, типы) — на **английском**.
- **Тексты в UI и ошибки пользователю** — только **английский** (constitution 8.2).

---

## 4. Ограничения (соблюдать строго)

- **Git:** не выполнять коммиты, push, pull, смену веток, rebase и т.п. без **явного указания** пользователя. По умолчанию результат — предложенный diff, не действия в репозитории.
- **Консоль:** команды выполняются в рабочей директории проекта; не добавлять лишние `cd` в каждую команду.
- **Минимальный дифф:** менять только то, что нужно для задачи; не делать широких рефакторингов и переименований «заодно».
- **Существующие утилиты:** перед новой функцией проверять `internal/debuglog`, `internal/constants`, `core/network_utils`, `internal/platform`, пакеты в `core/config`, `ui/wizard/business`.
- **Логирование:** только `internal/debuglog`; в новых путях кода — `debuglog.DebugLog` в точках start/success/error (см. **constitution.md**, раздел 5).
- **Ошибки пользователю:** только через `dialogs.ShowError` / `ShowErrorText`; не игнорировать ошибки (`_ = err` только с явным обоснованием в комментарии).
- **Информация об агентах:** без явного запроса пользователя не добавлять ни в документы, ни в коммиты, ни в код упоминания используемых агентов (Cursor, Codex, названия моделей и т.п.).

Полный список — в **constitution.md** (раздел «Запреты») и **IMPLEMENTATION_PROMPT.md** (в т.ч. разделы про Git и консоль).

## 5. Контракт выхода после реализации

После завершения реализации выдавай (кратко, в Markdown):

1. **План изменений** (1–3 пункта) — что сделано.
2. **Список изменённых файлов** — полные пути.
3. **Ключевые фрагменты кода** (с путями) — только существенные изменения.
4. **Команды для проверки:** `go build ./...`, `go test ./...`, `go vet ./...`, линтер при наличии.
5. **Риски и ограничения** — если есть; как проверить.
6. **Assumptions** — если были предположения из-за неполного ТЗ.

Не пересказывай ТЗ целиком, не вставляй большие простыни кода — только нужное для проверки и понимания.

## 6. Где что искать

- **Архитектура и потоки:** `docs/ARCHITECTURE.md`
- **Принципы и запреты:** `todo/constitution.md`
- **Текущие фичи и задачи:** `todo/README.md`, затем `todo/{FEATURE}/SPEC.md`, `PLAN.md`, `TASKS.md`
- **Правила реализации и DoD:** `todo/IMPLEMENTATION_PROMPT.md`
- **Сборка, тесты:** `docs/BUILD_WINDOWS.md`, `docs/TEST_README.md`; тесты — рядом с кодом (`*_test.go`), GUI-пакеты исключены из `go test` (см. constitution).
- **Конфиг парсера:** `docs/ParserConfig.md`; шаблон визарда — `bin/wizard_template.json`.

## 7. Краткий чеклист перед завершением

- [ ] Сборка и тесты проходят (`go build ./...`, `go test ./...`, `go vet ./...`).
- [ ] Соблюдён IMPLEMENTATION_PROMPT (логирование, ошибки, комментарии, минимальный дифф, без лишних абстракций).
- [ ] В новых путях кода есть debuglog.DebugLog при start/success/error.
- [ ] Нет действий с Git без явной просьбы пользователя.
- [ ] Изменены только файлы из TASKS.md; новые файлы — только если указаны в SPEC/PLAN.
