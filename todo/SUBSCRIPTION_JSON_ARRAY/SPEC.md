# Спецификация: Поддержка подписок в формате JSON-массива конфигов

## 1. Проблема

### 1.1 Текущие форматы подписки

Парсер подписок в singbox-launcher поддерживает только два формата ответа от subscription URL:

1. **Base64** — тело ответа в base64; после декодирования получается текст с построчными ссылками (`vless://...`, `vmess://...` и т.д.).
2. **Plain text** — тело ответа в виде обычного текста, по одной ссылке на строку.

Если сервер возвращает **JSON** (объект или массив), декодер явно отклоняет такой контент:

```go
// core/config/subscription/decoder.go
if strings.HasPrefix(contentStr, "{") || strings.HasPrefix(contentStr, "[") {
    return nil, fmt.Errorf("subscription URL returned JSON configuration instead of subscription list ...")
}
```

### 1.2 Реальный кейс

Некоторые провайдеры (например, tv123.ru и аналогичные панели) отдают по subscription URL **JSON-массив полных конфигов sing-box**: каждый элемент массива — отдельный конфиг с `log`, `dns`, `inbounds`, `outbounds`, `routing`, `remarks` и т.д. Пользователь вставляет такую ссылку в визард и получает ошибку: подписка не распарсилась.

### 1.3 Цель

Добавить поддержку **третьего формата** подписки: **ответ в виде JSON-массива конфигов**. Не заменять и не ломать существующие форматы (base64, plain text). После изменений одна и та же подпись (источники, визард, лимиты) должна обрабатывать все три формата в зависимости от фактического содержимого ответа.

## 2. Требования

### 2.1 Распознавание формата

- После успешной загрузки тела ответа (FetchSubscription) вызывается логика декодирования/разбора.
- Если тело после trim начинается с `[` (и является валидным JSON-массивом) — считать формат **«JSON array of configs»** и обрабатывать по новой ветке, **не** возвращая ошибку «JSON configuration instead of subscription list».
- Если тело начинается с `{` — по-прежнему считать одиночный JSON-конфиг **не** форматом подписки (ошибка или отдельная фича в будущем). В рамках этой задачи поддерживается только **массив**.
- Порядок проверок: сначала попытка base64; затем проверка на `[` и разбор как JSON-массив; затем проверка на `{` → ошибка; затем plain text (строки с `://`).

### 2.2 Разбор JSON-массива конфигов

- Каждый элемент массива — объект, совместимый с конфигом sing-box (наличие полей `outbounds`, при необходимости `remarks` и др.).
- Для **каждого** элемента массива:
  - Из поля `outbounds` (массив) взять только **прокси-аутбаунды**: типы `vless`, `vmess`, `trojan`, `shadowsocks`, `hysteria`, `hysteria2`, `ssh` и иные, которые лаунчер умеет использовать как ноды. **Исключать** служебные: `freedom`, `block`, `dns`, `blockhole` и т.п.
  - Для каждого такого аутбаунда сформировать одну **ноду** (ParsedNode) в общем списке нод данной подписки.
- Имя/тег ноды: из поля аутбаунда `tag`; при необходимости обеспечить уникальность в рамках подписки (например, префикс из `remarks` конфига при совпадении тегов между конфигами). Итоговый тег ноды не должен пустым; при отсутствии `tag` — сгенерировать (например, по индексу и remarks).

### 2.3 Интеграция в существующий поток

- Список нод, полученный из JSON-массива конфигов, передаётся в тот же pipeline, что и ноды из base64/plain text: применение TagPrefix/TagPostfix/TagMask, лимит MaxNodesPerSubscription, учёт Skip, MakeTagUnique и т.д. Поведение визарда и парсера конфига для такого источника не должно отличаться от обычной подписки.
- В визарде при проверке URL (CheckURL): если по ссылке пришёл JSON-массив конфигов, считать подписку валидной и показывать количество найденных нод (как для base64/plain text).
- При применении подписки (ApplyURLToParserConfig, LoadNodesFromSource) — те же шаги: одна запись в ProxySource с этим URL, при обновлении конфига — загрузка подписки и разбор в ноды; при сборке итогового конфига ноды из JSON-конфигов участвуют наравне с нодами из ссылок.

### 2.4 Обработка ошибок и граничных случаев

- Невалидный JSON (не массив, синтаксическая ошибка) — считать подписку некорректной: одна понятная ошибка пользователю (например, «subscription returned invalid JSON array»), без паники.
- Пустой массив `[]` — 0 нод, не ошибка (аналогично пустому списку ссылок).
- Элемент массива без `outbounds` или с не-массивом — пропускать элемент, не падать.
- Аутбаунд без поля `tag` — подставлять сгенерированный тег, чтобы нода всё равно попала в список (и была уникальной в рамках подписки).

### 2.5 Критерии приёмки

1. Подпись по URL, который возвращает **JSON-массив конфигов** (как tv123.ru), успешно обрабатывается: ошибка «JSON configuration instead of subscription list» больше не возвращается для ответов, начинающихся с `[`.
2. Из каждого конфига в массиве извлекаются только прокси-аутбаунды; каждый такой аутбаунд даёт одну ноду в общем списке подписки.
3. Количество нод ограничивается MaxNodesPerSubscription; применяются TagPrefix/TagPostfix/TagMask и уникальность тегов.
4. В визарде при проверке такой ссылки отображается успех и количество найденных нод; при применении подписка добавляется в ParserConfig и при следующей сборке/обновлении ноды из JSON-конфигов попадают в сгенерированный конфиг.
5. Форматы base64 и plain text продолжают работать без регрессий.
6. Сообщения об ошибках и логи — на английском (UI/пользователь) и по правилам проекта (debuglog в ключевых точках).

## 3. Структура данных

### 3.1 Вход: тело ответа подписки

- `content []byte` — сырое тело HTTP-ответа.
- После trim: если `contentStr[0] == '['` и `json.Valid(content)` — разбирать как JSON-массив.

### 3.2 Элемент массива (один конфиг)

- Ожидаемые поля (минимально): `outbounds` (массив объектов).
- Опционально: `remarks` (строка) — используется для уникальности тегов и логов.
- Каждый объект в `outbounds` с полем `type` (или `protocol` в зависимости от формата): если тип — прокси (vless, vmess, trojan, shadowsocks, hysteria, hysteria2, ssh, …), элемент учитывается; иначе (freedom, block, dns, blackhole, …) пропускается.

### 3.3 Выход: список нод

- Те же `[]*config.ParsedNode`, что и для base64/plain text.
- Для ноды, созданной из аутбаунда JSON-конфига: поле `Outbound` заполняется структурой этого аутбаунда (map[string]interface{}), поле `Tag` — выбранный или сгенерированный тег. Остальные поля ParsedNode (Server, Port, Scheme, …) при необходимости заполняются из аутбаунда для совместимости с GenerateNodeJSON (см. PLAN).

## 4. Ограничения и риски

- Поддерживается только **массив** конфигов (`[...]`). Одиночный объект конфига (`{...}`) в рамках этой задачи по-прежнему не считается форматом подписки.
- Глубокую валидацию схемы каждого конфига и каждого аутбаунда не делаем: достаточно наличия `outbounds` (массив) и у каждого аутбаунда — типа и тега (или генерации тега). Невалидные элементы пропускаются.
- Язык UI и сообщений пользователю — только английский (constitution).
- Логирование — только через `internal/debuglog`; в новых путях кода — вызовы в точках start/success/error.
