# SPEC: OUTBOUND_EDIT_PREVIEW_TAB — вкладка Preview в окне Edit Outbound

> Эта спецификация **продолжает работу**, начатую в `SPECS/008-F-C-OUTBOUNDS_CONFIGURATOR` (встроенный конфигуратор outbounds во вкладке Outbounds визарда и диалог Edit/Add Outbound).

## 1. Проблема

### 1.1 Текущее состояние

- Визард уже содержит:
  - вкладку `Sources` с Preview распарсенных узлов по всем источникам;
  - вкладки `Rules` / `Preview`, которые показывают результат применения ParserConfig в целом.
- Встроенный конфигуратор outbounds на вкладке **Outbounds** позволяет:
  - редактировать outbounds через диалог **Edit Outbound** (`Settings` + `Raw`);
  - задавать `Filters.tag`, `preferredDefault.tag` и `AddOutbounds` (direct-out / reject / другие теги).

При этом **нет локального превью для одного конкретного outbound**:

- Пользователь не видит, **какие именно узлы (теги нод)** попадут в этот outbound по заданным фильтрам.
- Неочевидно, **какая нода реально станет default** по `preferredDefault`.
- При работе с несколькими источниками и сложными паттернами (`!/regex/i`) трудно быстро проверить, что правило настроено правильно; приходится:
  - либо на глаз прикидывать соответствие паттернов и тегов;
  - либо собирать конфиг и смотреть косвенно по поведению.

### 1.2 Что нужно

Нужен **локальный Preview в окне Edit/Add Outbound**, который:

- показывает **список конкретных нод**, на которые подействует текущий outbound (с учётом `Filters` и `preferredDefault`);
- использует **ровно те же данные и те же функции/хелперы**, которые уже применяются при генерации итогового конфига и глобальных Preview (никакой «отдельной магии» или дублирования бизнес‑логики);
- не меняет поведение существующих вкладок визарда, а только добавляет удобную визуализацию для одного outbound.

---

## 2. Требования

### 2.1 Вкладка Preview в окне Edit Outbound

- В диалоге `ShowEditDialog` (окно Edit/Add Outbound) добавить **третью вкладку**:
  - существующие вкладки: `Settings`, `Raw`;
  - новая вкладка: **`Preview`**.
- Переключение вкладки **не должно** само по себе изменять ParserConfig или сохранять outbound; вкладка только отображает превью.
- Текст вкладок и всё содержимое UI — **на английском**:
  - заголовок вкладки: `Preview`;
  - вспомогательные подписи и кнопки: `Preview`, `Refresh preview`, `Matched nodes`, `Default node`, `Source`, и т.п.

### 2.2 Что именно показывает Preview

Preview строится **для текущего редактируемого outbound**, с учётом его настроек в форме:

- Источники данных:
  - актуальная структура `config.ParserConfig` из модели визарда (включая все Sources и outbounds);
  - **единый кеш распарсенных узлов (preview‑cache)** по всем источникам, который уже используется/будет использоваться вкладками `Sources` / `Rules` / `Preview`:
    - кеш заполняется и актуализируется в бизнес‑слое при изменениях ParserConfig;
    - все UI‑клиенты превью (Sources‑Preview, глобальный Preview, вкладка Preview в Edit Outbound) **только читают** из этого кеша и не триггерят парсинг напрямую.
- Используются **текущее состояние полей диалога**:
  - `Tag`, `Type`, `Filters.tag`, `PreferredDefault.tag`, `AddOutbounds` — как они сейчас заполнены в `Settings`;
  - если вкладка `Raw` была отредактирована и пользователь вернулся в `Settings` (или переключился на `Preview`), превью опирается на **актуальный outbound, показанный в `Settings`** (см. 2.3).

Отображаемая информация:

- **Общий список всех распарсенных нод**:
  - в таблицу/список попадают **все ноды (теги)**, которые есть в Preview по текущему ParserConfig (в том числе те, которые не попадают в outbound по фильтру);
  - для каждой строки отображаются минимум поля:
    - `Tag` (тег ноды);
    - `Source` — короткий label источника (`Global` или label из списка Sources);
    - (опционально) тип / провайдер узла, если это легко достать из уже существующих структур Preview.
  - Визуальная кодировка принадлежности:
    - ноды, которые **попадают в outbound по Filters.tag**, подсвечиваются **зелёным** (green foreground/фон в пределах принятых в приложении тем);
    - ноды, которые **не попадают в outbound**, подсвечиваются **красным** (red);
    - default‑нода (по `PreferredDefault.tag`) **всегда поднимается в начало списка** и выделяется **синим** (blue), при этом она тоже считается «попавшей» (зелёной) логически.
- **AddOutbounds в общем списке**:
  - элементы `AddOutbounds` (`direct-out`, `reject`, другие теги) отображаются в конце основного списка как отдельные строки;
  - для них можно использовать отдельный тип строки (например, префикс `[outbound]` в колонке `Tag`);
  - цвет по умолчанию — нейтральный (без зелёного/красного статуса), чтобы не смешивать их с нодами.

В эту спецификацию **не входит** глобальное представление всего конфига — Preview фокусируется только на одном outbound.

### 2.3 Связь между Settings / Raw и Preview

- Источником правды для Preview в рамках диалога является **структура outbound, эквивалентная тому, что сохранится при нажатии Save**, а не «сырые» поля по отдельности.
- Требования:
  - При открытии диалога:
    - исходное состояние outbound берётся из `existing` (или дефолтного `OutboundConfig` для Add);
    - `Settings`, `Raw` и `Preview` должны быть консистентны.
  - Вкладка `Raw` содержит:
    - многострочное поле с JSON представлением `OutboundConfig`;
    - и **кнопку/ссылку `Documentation`**, открывающую документацию по структуре outbound‑конфига в разделе `Секция outbounds` файла ParserConfig (URL: `https://github.com/Leadaxe/singbox-launcher/blob/main/docs/ParserConfig.md#%D1%81%D0%B5%D0%BA%D1%86%D0%B8%D1%8F-outbounds`), по аналогии с кнопкой Documentation для ParserConfigJSON на вкладке Outbounds.
  - При изменениях на вкладке `Settings`:
    - Preview использует значения из текущих виджетов (`tagEntry`, `filterValEntry`, `defValEntry`, чекбоксы AddOutbounds и т.п.), **без необходимости переключаться на Raw**.
  - При работе через `Raw`:
    - изменения в `Raw` **вступают в силу для Preview только после успешного парсинга**;
    - в этой задаче допустимо ограничиться тем, что Preview опирается на **форму `Settings`**; синхронизация «Raw → Settings» может остаться как есть (по текущей логике сохранения).
- Спецификация **явно не требует** live‑синхронизации всех трёх вкладок; достаточно, чтобы:
  - Preview показывал корректный результат для тех данных, которые реально будут сохранены при `Save`;
  - при очевидных несоответствиях (невалидный JSON на Raw) Preview не ломался и показывал понятное сообщение об ошибке (см. 2.5).

### 2.4 Обновление Preview

- При **переключении на вкладку `Preview`**:
  - **сразу (без задержек и без необходимости нажимать кнопки)**:
    - собирается текущая «виртуальная» конфигурация outbound’а из формы;
    - к распарсенным узлам из preview‑cache применяются те же фильтры, что и при построении реального outbound;
    - UI списка (все ноды с зелёным/красным статусом и выделенной синей default‑нодой) и сводка по AddOutbounds обновляются.
- **Автообновления при вводе на вкладке `Settings` или в `Raw` нет**:
  - изменения в форме попадают в превью **только**:
    - при повторном входе на вкладку `Preview` (после переключения с другой вкладки диалога);

### 2.5 Ошибки и поведение при некорректных данных

- Если при построении превью возникает ошибка:
  - некорректный паттерн в `Filters.tag` или `PreferredDefault.tag`;
  - внутренние ошибки фильтрации или доступа к распарсенным нодам;
  - невалидный JSON в Raw (если он влияет на текущий outbound);
- то:
  - на вкладке Preview **не падает паника** и не ломается окно;
  - вместо списка показывается человекочитаемое сообщение (на английском), например:
    - `Failed to build preview: invalid filter pattern` или
    - `Failed to build preview: invalid outbound JSON. Please switch to the "Raw" tab and fix JSON first.`;
  - сообщения об ошибке в диалоговом окне через `dialogs.ShowError` **не требуются**; достаточно inline‑ошибки в контенте Preview;
  - при ошибке, связанной с Raw JSON, Preview **явно подсказывает пользователю перейти на вкладку "Raw" и исправить JSON**, без попыток автоматически модифицировать или чинить его.

### 2.6 Ограничения по производительности и объёму

- Preview должен быть:
  - ограничен по высоте (scroll), чтобы окно Edit Outbound не разрасталось вертикально;
  - способен отображать десятки/сотни нод без заметных лагов при единичном `Refresh`.
- Для предотвращения перегрузки:
  - допустимо:
    - ограничить количество одновременно отображаемых строк (например, `first N, then “and X more…”`);
    - добавить простую сортировку (по Tag или Source) — опционально.
  - В рамках этой спецификации минимальное требование — **корректность и отсутствие зависаний** при разумном количестве нод; сложная навигация по списку не обязательна.

### 2.7 Кеш превью (общий для всех Preview)

- Должен быть реализован **единый кеш распарсенных нод** для всего визарда:
  - хранит результат парсинга Sources (узлы, их теги, связь с источниками и т.п.);
  - обновляется только в одном месте — в бизнес‑логике при изменениях ParserConfig / Sources;
  - инвалидация/пересчёт по уже существующим событиям (AppendURLsToParserConfig, правки Outbounds, ручной JSON и т.д.).
- Все виды превью используют **один и тот же кеш**:
  - блок Preview на вкладке `Sources`;
  - глобальные вкладки `Rules` / `Preview`;
  - новая вкладка `Preview` в диалоге Edit/Add Outbound.
- UI‑слой (в т.ч. Outbound Edit Preview) не должен самостоятельно запускать тяжёлый парсинг; он запрашивает данные только из кеша и, при необходимости, может лишь попросить бизнес‑слой «обновить кеш» через уже существующие async‑механизмы.

## 3. Критерии приёмки

1. В окне Edit/Add Outbound появляется третья вкладка `Preview` рядом с `Settings` и `Raw`; все подписи — на английском.
2. При переключении на вкладку Preview автоматически отображается актуальный список `Matched nodes`:
   - каждая строка содержит минимум `Tag` и `Source` (Global или label источника);
   - default‑нода по `PreferredDefault.tag` визуально выделена.
3. В нижней части вкладки (или над списком) отображается сводка по `AddOutbounds` вида `Also includes: ...`; при отсутствии дополнительных outbounds список явно пустой или строка скрыта.
4. Preview не изменяет ParserConfig и не сохраняет outbound сам по себе; после закрытия диалога без `Save` никаких изменений в конфиге не остаётся.
5. При некорректных фильтрах или несогласованных данных Preview не падает: контент вкладки показывает понятное текстовое сообщение об ошибке, окно Edit Outbound остаётся рабочим.
6. При типичном количестве источников и нод обновление превью по `Refresh` происходит без заметных лагов (окно не «замерзает» на несколько секунд).
