## PLAN: OUTBOUND_EDIT_PREVIEW_TAB — вкладка Preview в Edit Outbound

### 1. Общая идея

- Добавить в окно `Edit/Add Outbound` третью вкладку **Preview**, которая показывает список всех распарсенных нод с подсветкой:
  - зелёным — ноды, попавшие под фильтр текущего outbound;
  - красным — ноды, не попавшие под фильтр;
  - синим и вверху списка — default‑нода по `preferredDefault`.
- Не придумывать новую логику: **использовать те же структуры и функции**, которые уже применяются при генерации конечного конфига и существующих Preview (Sources/Rules/Preview).
- Вынести в бизнес‑слой **единый кеш распарсенных нод (preview‑cache)**, которым будут пользоваться:
  - Preview на вкладке `Sources`;
  - окна `View` по источникам;
  - глобальные вкладки `Rules` / `Preview`;
  - новая вкладка `Preview` в Edit Outbound.

### 2. Компоненты и изменения по файлам

#### 2.1 Бизнес‑слой и модель визарда

- **Где:** `ui/wizard/business/*`, `ui/wizard/models/wizard_model.go`, при необходимости — вспомогательные типы в `core/config`.
- **Что сделать:**
  - Ввести структуру / тип кеша preview‑данных (например, «список всех нод с метаданными: тег, источник, тип, технические поля»).
  - Добавить в модель визарда поле для хранения этого кеша (например, `PreviewNodesCache` или аналог), доступное презентеру и UI.
  - Реализовать функции бизнес‑слоя:
    - «пересчитать кеш по текущему `ParserConfig`» — используя уже существующие функции, которые сейчас применяются для генерации Preview в Sources / Rules / Preview;
    - «получить все ноды для Preview» (общий список для вкладки Sources и Outbound‑Preview);
    - «получить ноды по конкретному источнику» (для окон View).
  - В местах, где сейчас генерируется Preview «в лоб» (Sources‑Preview, окна View, глобальные Preview), переключить использование на новый кеш:
    - парсинг/разбор выполняется один раз в бизнес‑слое;
    - UI‑слой только читает готовые данные.

#### 2.2 Вкладка Preview в Edit Outbound (UI)

- **Где:** `ui/wizard/outbounds_configurator/edit_dialog.go`.
- **Что сделать:**
  - Расширить `ShowEditDialog`:
    - добавить третью вкладку `Preview` в `container.NewAppTabs` рядом с `Settings` и `Raw`;
    - при выборе вкладки `Preview` вызывать хелпер, который:
      - собирает «виртуальный» `OutboundConfig` из текущих полей формы (`Settings`) или, при необходимости, из Raw JSON (с обработкой ошибок парсинга);
      - берёт из preview‑cache общий список нод;
      - помечает ноды как «попавшие/не попавшие» по тем же фильтрам, что используются в реальной генерации outbound;
      - определяет default‑ноду по `preferredDefault`.
  - UI вкладки Preview:
    - скроллируемый список всех нод, отсортированный так, чтобы default‑нода была первой;
    - визуальная подсветка:
      - зелёный (ok) — попавшие ноды;
      - красный — не попавшие;
      - синяя подсветка для default‑ноды (поверх зелёного статуса);
    - текстовая сводка по `AddOutbounds` внизу или сверху списка.
  - При ошибке парсинга Raw JSON:
    - не строить список;
    - выводить понятное текстовое сообщение с рекомендацией перейти на вкладку `Raw` и исправить JSON.

#### 2.3 Вкладка Raw: ссылка на документацию

- **Где:** `ui/wizard/outbounds_configurator/edit_dialog.go`.
- **Что сделать:**
  - Во вкладке `Raw` над/под многострочным полем JSON добавить кнопку/ссылку `Documentation`.
  - Эта кнопка должна открывать документацию по структуре outbound‑конфига — раздел **«Секция outbounds»** в `docs/ParserConfig.md`:
    - URL: `https://github.com/Leadaxe/singbox-launcher/blob/main/docs/ParserConfig.md#%D1%81%D0%B5%D0%BA%D1%86%D0%B8%D1%8F-outbounds`.
  - Реализовать открытие так же, как уже сделано для кнопки Documentation на вкладке Outbounds (ParserConfigJSON), чтобы поведение было единообразным.

#### 2.4 Интеграция с существующими Preview

- **Где:** файлы, отвечающие за:
  - Preview на вкладке `Sources` (список распарсенных серверов);
  - окна `View` по источникам;
  - вкладки `Rules` / `Preview` визарда.
- **Что сделать:**
  - Перевести существующие места, где создаются Preview, на использование общего preview‑cache:
    - убрать/минимизировать дублирующий парсинг;
    - убедиться, что данные и поведение остаются теми же (те же фильтры, та же сортировка, то же отображение).
  - Проверить, что обновление кеша триггерится:
    - при изменении Sources (AppendURLsToParserConfig, удаление источников, смена префиксов и т.д.);
    - при изменениях в outbounds / ParserConfig JSON из вкладки Outbounds;
    - при механизмах, которые уже сейчас запускают Parse/Preview (TriggerParseForPreview и пр.).

### 3. Тестирование и UX‑проверка

- Проверить сценарии:
  - исходное открытие Edit Outbound без изменений (Preview корректен);
  - изменение фильтра `Filters.tag` и переход на вкладку Preview;
  - настройка `preferredDefault` и проверка выделения default‑ноды;
  - изменение `AddOutbounds` и отображение сводки;
  - некорректный JSON на вкладке Raw — Preview показывает текст ошибки и предлагает вернуться в Raw.
- Пройтись по типичным конфигам (несколько источников, много нод, сложные фильтры) и убедиться, что:
  - Preview не «фризит» окно;
  - данные совпадают с тем, что реально попадает в outbound в итоговом конфиге.
