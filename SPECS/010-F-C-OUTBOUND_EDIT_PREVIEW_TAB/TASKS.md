## TASKS: OUTBOUND_EDIT_PREVIEW_TAB

### Этап 1. Общий preview‑кеш в бизнес‑слое

- [ ] Ввести структуру данных для preview‑кеша (список распарсенных нод с тегом, источником и вспомогательными полями).
- [ ] Добавить поле кеша в модель визарда (wizard model), доступное презентеру и UI.
- [ ] Реализовать бизнес‑функции:
  - [ ] пересчёт кеша по текущему `config.ParserConfig` (с опорой на существующие функции парсинга/Preview);
  - [ ] получение всех нод для общего Preview (Sources, Outbound‑Preview);
  - [ ] получение нод для конкретного источника (View по `ProxySource`).
- [ ] Встроить обновление кеша в существующий поток:
  - [ ] после AppendURLsToParserConfig / изменения Sources;
  - [ ] после операций в конфигураторе outbounds (Edit/Add/Del/Up/Down + ручной JSON);
  - [ ] в местах, где уже триггерится Parse/Preview (TriggerParseForPreview и аналоги).

### Этап 2. Перевод существующих Preview на кеш

- [ ] Вкладка `Sources`:
  - [ ] использовать preview‑кеш как источник данных для списка распарсенных серверов;
  - [ ] убедиться, что UI и поведение (фильтры, сортировка) не изменились.
- [ ] Окна `View` по источникам:
  - [ ] получать ноды для конкретного источника из кеша;
  - [ ] не запускать свой парсинг.
- [ ] Вкладки `Rules` / `Preview`:
  - [ ] при необходимости использовать кеш там, где сейчас повторно разбираются ноды;
  - [ ] убедиться, что итоговый набор outbounds и Preview совпадает с текущей реализацией.

### Этап 3. Вкладка Preview в Edit/Add Outbound

- [ ] В `ShowEditDialog` добавить третью вкладку `Preview` рядом с `Settings` и `Raw`.
- [ ] При входе на вкладку `Preview`:
  - [ ] собрать «виртуальный» `OutboundConfig` из текущих полей `Settings` (Tag, Type, Filters, PreferredDefault, AddOutbounds);
  - [ ] использовать preview‑кеш для получения полного списка нод;
  - [ ] применить те же фильтры, что и реальная генерация outbound, чтобы разделить ноды на попавшие/не попавшие;
  - [ ] вычислить default‑ноду по `PreferredDefault.tag` и поднять её в начало списка.
- [ ] UI вкладки:
  - [ ] скроллируемый список всех нод:
    - [ ] Tag, Source (Global / label источника), при желании тип/провайдер;
    - [ ] зелёная подсветка для нод, попавших в outbound;
    - [ ] красная подсветка для нод, не попавших в outbound;
    - [ ] синяя выделенная строка default‑ноды (сверху списка).
  - [ ] текстовая сводка `AddOutbounds` (в конце/начале списка), формат вида `Also includes: direct-out, reject, tag1, ...`.
- [ ] Обработка ошибок:
  - [ ] при некорректных фильтрах или внутренних ошибках показывать текст ошибки вместо списка;
  - [ ] при невалидном JSON на вкладке Raw — сообщение `Failed to build preview: invalid outbound JSON. Please switch to the "Raw" tab and fix JSON first.` без попытки автоисправления.

### Этап 5. Тесты и регрессия

- [ ] Ручное тестирование типичных сценариев:
  - [ ] несколько источников, разные префиксы/маски тегов;
  - [ ] сложные фильтры с `!/regex/i` и несколькими ключами (tag/host и т.п.);
  - [ ] проверка default‑ноды и AddOutbounds.
- [ ] Проверить, что:
  - [ ] Preview не вызывает заметных лагов при десятках/сотнях нод;
  - [ ] данные на вкладке Preview соответствуют реальному содержимому outbound в сгенерированном конфиге;
  - [ ] существующие Preview (Sources / Rules / Preview / View) ведут себя так же, как до изменений.
