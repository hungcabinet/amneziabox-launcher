# Спецификация: Правила Processes — режим по пути (path based)

## 1. Проблема

### 1.1 Текущее правило «Processes»

В диалоге «Add Rule» при выборе типа **Processes** пользователь выбирает процессы по **имени** (`process_name`): список исполняемых имён из списка запущенных процессов. Нет возможности задать правило по **пути** исполняемого файла (например «все, у кого в пути есть `steam`»), без ручного редактирования JSON.

### 1.2 Цель

- Оставить **один** тип правила **Processes**.
- Добавить в режим Processes опцию **«Match by path»** (чекбокс `[ ] Match by path`).
- **Снят** — текущее поведение: выбор процессов по имени (`process_name`).
- **Установлен** — показывать поле ввода строк (шаблоны пути) и переключатель **Simple** / **Regex**. В режиме Simple: `*` при сохранении заменяется на `(.*)`. В режиме Regex: строки трактуются как готовые regex и записываются в **`process_path_regex`** как есть (с проверкой валидности).

---

## 2. Требования

### 2.1 Расширение UI типа Processes

- Выбор типа правила: **четыре чекбокса** (IP Addresses (CIDR), Domains/URLs, Processes, Custom JSON). Всегда выбран ровно один тип; повторное нажатие на выбранную галочку не меняет выбор. При открытии диалога **создания** по умолчанию выбрана первая позиция (IP); при **редактировании** — тип из сохранённого правила.
- В строке с чекбоксом **Processes** по центру — чекбокс **«Match by path»** (в одной строке с Processes).
- **Match by path снят (по умолчанию):**
  - Видны: подпись «Processes (select one or more via popup):», кнопка «Select Processes...», список выбранных имён. Сохранение — в `process_name` (массив имён), как сейчас.
- **Match by path включён:**
  - Скрыть кнопку «Select Processes...» и список выбранных имён.
  - Показать переключатель **Simple** / **Regex** и многострочное поле ввода (одна строка — один шаблон). Placeholder поля меняется при переключении Simple/Regex; для Regex — указание, что ввод «как есть», без обёртки вида /regex/i.
  - **Simple:** подсказка: можно использовать `*` как «любая последовательность»; при сохранении `*` → `(.*)`, остальные символы экранируются для regex. Примеры: `*/steam/*`, `*\\Steam\\*`, `C:\Games\*`.
  - **Regex:** подсказка: готовые регулярные выражения; при сохранении строки записываются в `process_path_regex` без преобразований, только проверка валидности (`regexp.Compile`).
  - Результат в обоих режимах — массив regex-строк в поле **`process_path_regex`** (без `process_name`).
- В одном правиле типа Processes используется **либо** `process_name`, **либо** `process_path_regex` — в зависимости от чекбокса. Режим Simple/Regex в state не сохраняем: в rule всегда массив regex; при открытии на редактирование показываем сохранённые строки, переключатель — **Regex**.

### 2.2 Преобразование и валидация при сохранении

- **Simple:** для каждой непустой строки заменить `*` → `(.*)`, остальные спецсимволы regex в литералах экранировать (чтобы строка была валидным regex). Конкретный алгоритм — в PLAN.
- **Regex:** строки не менять; только проверить, что каждая непустая строка компилируется в regex (`regexp.Compile`).
- Валидация: при включённом Match by path — хотя бы одна непустая строка; после преобразования (Simple) или как есть (Regex) каждая строка должна быть валидным regex; при ошибке — сообщение пользователю, не сохранять.

### 2.3 Конфиг и состояние

- В конфиге: при Match by path в правило попадает **`process_path_regex`** (массив regex-строк); при обычном режиме — **`process_name`** (массив имён). Outbound и остальное — как у существующих правил.
- В state (PersistedCustomRule) тип правила остаётся **«Processes»**. В **rule** храним либо `process_name`, либо `process_path_regex` (массив regex). Режим Simple/Regex в state не сохраняем. При загрузке: если в rule есть `process_path_regex` — выставить чекбокс Match by path, в многострочное поле подставить сохранённые regex-строки, переключатель установить в **Regex** (пользователь видит фактическое содержимое rule). Обратное отображение «regex → Simple с *» при редактировании — опционально.

### 2.4 Критерии приёмки

1. В режиме Processes есть чекбокс «Match by path»; при его включении — переключатель Simple / Regex и многострочное поле ввода (подсказки: Simple — про `*`, Regex — про готовые regex).
2. При сохранении с Match by path: в режиме Simple — `*` → `(.*)` и экранирование; в режиме Regex — строки как есть, только проверка валидности; в rule записывается `process_path_regex` (массив); в конфиге — одно правило с `process_path_regex` и выбранным outbound.
3. При сохранении без Match by path поведение как сейчас: `process_name`, кнопка выбора процессов.
4. Загрузка/редактирование: правило с `process_path_regex` открывается с включённым Match by path, переключатель Regex, поле заполнено сохранёнными строками из rule.

## 3. Структура данных

### 3.1 Rule (конфиг и state)

- Тип правила в UI и в state — по-прежнему **Processes** (отдельный тип не вводится).
- **Обычный режим:** `rule` = `{ "process_name": ["name1", ...], "outbound": "..." }`.
- **Match by path:** `rule` = `{ "process_path_regex": ["<regex1>", ...], "outbound": "..." }`.
- В одном rule не смешивать `process_name` и `process_path_regex`.

### 3.2 DetermineRuleType

- Если в rule есть поле `process_path_regex` (массив), при загрузке тип определяем как **Processes** (как и при `process_name`). В диалоге редактирования по наличию `process_path_regex` выставляем чекбокс Match by path и заполняем поле шаблонов.

## 4. Ограничения

- **Платформа:** `process_path_regex` в sing-box поддерживается на Linux, Windows, macOS.
- **Язык UI:** тексты и сообщения об ошибках — только английский (CONSTITUTION).
