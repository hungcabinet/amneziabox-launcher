# Спецификация: локализация интерфейса (выбор языка)

Задача: добавить возможность выбора языка программы и перевести весь интерфейс, включая вкладку «Правила» визарда и длинные тексты (подсказки, описания, сообщения в диалогах).

## Проблема

Сейчас все тексты в UI захардкожены на английском (constitution 8.2, 8.3). Пользователи из РФ/СНГ и других регионов могут предпочитать интерфейс на родном языке. Нет единого механизма для:

- выбора языка приложения;
- хранения переводимых строк (коротких и длинных);
- подстановки переведённого текста в виджеты и диалоги.

Особенно важно поддержать **длинные текстовые константы**: во вкладке Rules визарда используются описания правил (Rule.Description из шаблона), подсказки (например, TUN: «Enabling TUN will require entering your password…»), блоки текста в диалогах (Sing-box, справка по правилам, ошибки). Их нельзя хранить в коде в виде множества коротких ключей — нужна схема, позволяющая держать многострочные и длинные тексты в переводах без потери читаемости и без дублирования логики.

## Цель

1. **Выбор языка** — пользователь может выбрать язык интерфейса (как минимум English и Русский); выбор сохраняется между сеансами.
2. **Полный охват UI** — переведены все элементы: главное окно (вкладки Core, Config, Servers, Diagnostics, Help), визард (Sources, Rules, Outbounds, Preview), диалоги (Add Rule, Load/Save state, Get free VPN, ошибки, информация), кнопки, метки, заголовки.
3. **Длинные тексты** — единый подход к переводу длинных констант: подсказки, описания в диалогах, справка (в т.ч. вкладка Rules: «Selectable rules», «Final outbound», текст в кнопке «?» для TUN и для правил из шаблона). Тексты из шаблона (Rule.Label, Rule.Description в wizard_template.json) могут оставаться в данных шаблона и не переводиться автоматически в рамках этой задачи, либо допускается отдельный механизм (например, ключи вида `rule.desc.<id>` в переводах) — уточнить в PLAN.
4. **Минимальная зависимость** — по возможности использовать стандартный подход (например, `golang.org/x/text` или один выбранный пакет), без тяжёлой инфраструктуры; строки — в отдельном слое (пакет/папка переводов), не размазаны по UI-коду.

## Требования

### 1. Выбор языка и хранение

- Выбранный язык хранится в файле настроек лаунчера **`bin/settings.json`** (путь: `GetBinDir(ExecDir)/settings.json`). Формат — JSON-объект, ключ языка: `"lang": "en"` / `"lang": "ru"` (ISO 639-1). Не в config.json sing-box.
- Язык по умолчанию — English (соответствует текущему поведению и constitution 8.2/8.3).
- **Где выбирать язык:** элемент выбора языка (кнопка, выпадающий список или подменю) размещается **во вкладке Help** (контент вкладки «❓ Help»). При смене языка UI обновляется без перезапуска приложения (см. п. 4).

### 2. Слой переводов

- Все строки, показываемые пользователю в UI (кнопки, метки, заголовки окон, сообщения об ошибках, тексты в диалогах), получаются через единый слой (например, функция `T(key string) string` или пакет `i18n`/`locale`).
- Ключи — осмысленные и устойчивые (например, `core.start`, `wizard.rules.final_outbound`, `dialog.add_rule.title`).
- Поддержка плейсхолдеров для подстановки (например, «Template file %s not found» → «Файл шаблона %s не найден»).

### 3. Длинные текстовые константы

- Длинные тексты (абзацы, многострочные подсказки, описания в ShowInformation/ShowCustom) не захардкожены в коде, а вынесены в каталог переводов с ключами.
- Формат хранения длинных текстов должен быть удобным для редактирования и перевода: отдельные файлы по языкам (JSON, YAML, или встроенные map в Go — см. PLAN), с возможностью многострочных значений (raw string, массив строк, или экранирование переносов).
- Примеры длинных текстов: описание TUN при нажатии «?» на вкладке Rules; текст в диалоге «Sing-box» (про «Extract the binary into the bin folder» и т.д.); сообщения типа «Save operation already in progress... Please wait.»; справка по «Selectable rules» / «Final outbound» при нажатии «?».

### 4. Обновление UI при смене языка

- При смене языка (из вкладки Help) интерфейс обновляется без обязательной перезагрузки приложения: перестраиваются виджеты с новыми строками или обновляются тексты там, где это технически возможно без полной пересборки окна (уточнить в PLAN в зависимости от Fyne).

### 5. Охват

- Переведены: Core (Core Status, Start, Stop, Exit, Config, Sing-box, кнопки и сообщения), визард (вкладки Sources, Rules, Outbounds, Preview; кнопки Add Rule, Outbound, Final outbound, SRS, «?», «✏️», «❌»; сообщения «No selectable rules…», «Template file … not found» и т.д.), диалоги (Add Rule, Load/Save state, Get free, ошибки через ShowError/ShowErrorText/ShowInformation), Diagnostics, Help.
- Константы из `internal/constants` (имена файлов, URL), логи (debuglog) и комментарии в коде не переводятся.

## Критерии приёмки

- [x] Во вкладке Help есть элемент выбора языка; выбранный язык сохраняется в `bin/settings.json` и применяется при следующем запуске.
- [x] По умолчанию используется английский язык.
- [x] Все видимые пользователю строки в UI (включая вкладку Rules визарда и диалоги) берутся из слоя переводов по ключам.
- [x] Длинные тексты (подсказки, описания в диалогах, справка) вынесены в переводимые константы с возможностью удобного редактирования многострочного текста.
- [x] Реализованы как минимум локали: English (en) и Русский (ru). Также добавлена поддержка дополнительных языков через `bin/locale` и загрузку по кнопке Help.
- [x] Сборка и существующие тесты проходят; линтер без новых замечаний. В новых путях кода — debuglog при необходимости.
- [x] Документация: в docs/release_notes/upcoming.md и при необходимости в docs/ARCHITECTURE.md отражено добавление локализации; constitution 8.2/8.3 допускает «язык по умолчанию — английский, при выборе языка отображается выбранная локаль».

## Замечания и границы

- **Шаблон визарда (wizard_template.json):** поля `label` и `description` правил — данные шаблона. В рамках задачи можно не переводить их автоматически (остаются в том языке, в котором заданы в шаблоне); при желании в будущем можно ввести ключи вида `rule.<id>.label` / `rule.<id>.description` в каталоге переводов — в SPEC не закладываем, при необходимости описать в PLAN как опцию.
- **Производительность:** загрузка переводов при старте приложения не должна блокировать UI (допустима синхронная загрузка небольшого объёма строк при старте; при больших каталогах — ленивая загрузка по языку).
- **Архитектура:** слой переводов не должен нарушать разделение UI и бизнес-логики; предпочтительно один пакет (например, `internal/locale` или `ui/locale`), от которого зависят только UI-пакеты.

## Решения по уточнениям

- **Виджет во вкладке Help:** выпадающий список (Select) «Language: [English ▼]» — достаточно; отдельный диалог не нужен.
- **Формат settings.json:** заложить расширяемую структуру с самого начала: один JSON-объект, например `{ "lang": "en" }`. В PLAN — чтение/запись только нужных полей, при добавлении новых настроек (тема, окно и т.д.) формат не менять, только дополнять поля.

## Зависимости

- Нет жёсткой зависимости от других открытых issues или фич. После реализации при необходимости обновить constitution 8.2/8.3 (формулировка про «язык по умолчанию» и выбор языка).
