# Спецификация: Данные лаунчера на macOS в системных каталогах (Application Support / Logs)

## 1. Проблема

### 1.1 Текущее поведение

- `ExecDir` определяется как каталог исполняемого файла (`filepath.Dir(os.Executable())`).
- На macOS при запуске из `.app` это **`…/singbox-launcher.app/Contents/MacOS/`**.
- Конфигурация, шаблоны, скачанный `sing-box`, rule-sets, `settings.json` и логи лежат **рядом с бинарником** (`bin/`, `logs/` под `Contents/MacOS/`).
- Такой **portable** layout удобен для единообразия с Windows/Linux и для «папки с программой», но:
  - полная замена `.app` (ошибочная или намеренная) **стирает пользовательские данные**;
  - это расходится с ожиданиями пользователей macOS (данные живут вне бандла, в `~/Library/…`).

### 1.2 Последствия

- Риск потери `config.json`, скачанных артефактов и логов при обновлении/копировании всего бандла.
- Сложнее объяснять путь к конфигу на Mac («внутри .app» вместо привычного Application Support).

---

## 2. Цели и границы

### 2.1 Цели

- На **macOS** при запуске из **GUI `.app`** (типичный сценарий `/Applications`):
  - хранить **пользовательские данные** (как минимум: `bin/` с `config.json`, шаблонами, скачанным ядром, rule-sets, `settings.json`) в каталоге в духе **`~/Library/Application Support/<идентификатор>/`**;
  - хранить **логи** в **`~/Library/Logs/<идентификатор>/`** (или согласованный с этим подкаталог внутри Application Support — зафиксировать в реализации в `PLAN.md`, не смешивать без причины).
- **Идентификатор приложения (Bundle ID)** для путей и для **`CFBundleIdentifier` в `Info.plist`**:
  - **значение по умолчанию:** `com.singbox-launcher`;
  - **должно быть изменяемым** одним согласованным способом (например константа в коде + тот же идентификатор в скрипте сборки `build_darwin.sh` через переменную окружения или параметр, либо `-X` при `go build`), без расхождения между plist и путями на диске.
- **Windows и Linux:** поведение **без изменений** — по-прежнему portable layout относительно `ExecDir` (если позже понадобится unification — отдельная задача).

### 2.2 Вне рамок (на этапе постановки)

- Notarization, подпись кода, hardened runtime — только если явно потребуется для доступа к `~/Library` (уточнить при реализации).
- Изменение модели данных или формата `config.json`.
- Sandbox / Mac App Store.

## 3. Требования

### 3.1 Платформа и режим

- Логика «системные каталоги» включается **только на macOS** и **только** когда приложение запущено как **собранный `.app`** (критерий — см. `PLAN.md`: надёжная эвристика, например суффикс пути `*.app/Contents/MacOS` или эквивалент).
- Запуск **бинарника из произвольной папки** (разработка, `go run`, CI) — **portable** поведение как сейчас (данные рядом с exe), чтобы не засорять `~/Library` при локальной отладке.

### 3.2 Пути и содержимое

- Под `Application Support/<идентификатор>/` размещается то, что сейчас логически относится к **`GetBinDir` / `EnsureDirectories`**: каталог `bin/` (включая `config.json`, `sing-box`, шаблоны, `rule-sets`, `settings.json` и т.д.) — либо тот же относительный layout, что и сейчас, но с корнем в Application Support (деталь в `PLAN.md`).
- Логи — в `~/Library/Logs/<идентификатор>/` **или** согласованное решение в `PLAN.md` с обоснованием (единый корень vs раздельные каталоги).

### 3.3 Bundle ID (изменяемый)

- Единая константа/источник правды для:
  - строки в **`Info.plist`** (`CFBundleIdentifier`);
  - сегмента пути в `Application Support` и `Logs`.
- **По умолчанию:** `com.singbox-launcher`.
- Сборка (`build_darwin.sh` и при необходимости CI) должна подставлять **тот же** идентификатор, что и в коде, без ручного дублирования в двух местах без синхронизации (предпочтительно: одна переменная в скрипте + `-X` в ldflags **или** генерация plist из шаблона).

### 3.4 Миграция (желательно)

- Если при первом запуске новой логики обнаружены данные в старом месте (`Contents/MacOS/bin/` и т.д.) — **опционально** (обязательность уточнить в `PLAN.md`): однократное копирование/перенос в Application Support с логированием в debuglog, чтобы пользователи после обновления не теряли конфиг.

### 3.5 Документация

- Обновить **README** (EN/RU): где на macOS лежат конфиг и логи в режиме `.app`.
- При необходимости — **docs/ARCHITECTURE.md** (поток данных, `FileService`).

## 4. Критерии приёмки

- На macOS из установленного `.app` файлы пользователя **не** зависят от перезаписи бинарника в `Contents/MacOS/` (обновление только exe не трогает данные; полная замена `.app` без удаления `~/Library` **сохраняет** данные пользователя).
- На Windows/Linux пути и поведение **как до изменений** (регрессионные проверки).
- Bundle ID по умолчанию `com.singbox-launcher`; смена идентификатора через согласованный механизм сборки даёт согласованные plist и пути.
- `go build ./...`, `go test ./...`, `go vet ./...` проходят (с учётом исключений GUI из тестов по CONSTITUTION).
