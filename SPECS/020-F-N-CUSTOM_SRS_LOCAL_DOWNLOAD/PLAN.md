# План: Локальное скачивание пользовательских правил «SRS»

## 1. Архитектура и подход

### 1.1 Переиспользование существующей логики SRS

- Найти и переиспользовать реализацию из задачи `004-F-C-SRS_LOCAL_DOWNLOAD`:
  - функция(и) скачивания SRS (например `DownloadSRS`), работа с `{ExecDir}/bin/rule-sets/`;
  - определение состояния кнопки `⬇/🔄/✔️` для встроенных правил;
  - интеграция с UI вкладки Rules (`ui/wizard/tabs/rules_tab.go` и связанные модели).
- Вынести общие части в единый слой (если ещё не вынесено):
  - определение пути к `{ExecDir}/bin/rule-sets/` (через `FileService` / `platform.ExecDir`);
  - проверка наличия локального SRS по tag;
  - запуск скачивания по URL → сохранение в `{tag}.srs`.

### 1.2 Единый механизм для встроенных и пользовательских SRS

- Ввести «универсальное представление» правила SRS для вкладки Rules:
  - источник: либо `selectable_rule` из шаблона, либо `PersistedCustomRule` типа `srs`;
  - список tag + URL (для пользовательских уже есть в 014/018);
  - флаг «isCustom».
- На основе этого представления:
  - вычислять состояние кнопки (`⬇/🔄/✔️`) одинаково для обоих видов правил;
  - вызывать один и тот же путь скачивания (разница только в источнике URL/тегов).

### 1.3 Генерация конфига

- Проверить текущую реализацию `create_config.go` / `MergeRouteSection`:
  - как сейчас подставляются `type: "local"` / `path` для встроенных SRS;
  - как генерируются remote rule_set для пользовательских SRS из 014/018.
- Расширить генерацию так, чтобы:
  - для любого rule_set (встроенного или пользовательского) по tag:
    - если `bin/rule-sets/{tag}.srs` существует → `type: "local"`, `path`;
    - иначе использовать remote URL (для пользовательских — из state, для встроенных — из шаблона).
- При этом структура wizard state не меняется: remote информация остаётся источником правды, локальное наличие определяется по файловой системе.

## 2. Изменения по файлам

### 2.1 Бизнес-логика скачивания SRS

- **Кандидатные файлы** (уточнить по коду):
  - `ui/wizard/business/srs_download.go` (если уже есть отдельный модуль);
  - либо функции в `ui/wizard/business/loader.go` / `create_config.go`.
- Задачи:
  - вынести общие операции:
    - `HasLocalSRS(tag string) (bool, error)` — проверка наличия `{ExecDir}/bin/rule-sets/{tag}.srs`;
    - `DownloadSRSGroup(ctx, entries []SRSEntry) error` — скачивание набора `(tag, url)` по аналогии с правилом «Russian blocked resources»;
  - обеспечить, чтобы эти функции не зависели от того, встроенное правило или пользовательское.

### 2.2 Модели wizarda (state/custom rules)

- Файлы (по 014/018):
  - `ui/wizard/models/wizard_state_file.go`
  - возможно, дополнительные структуры в `ui/wizard/models/*.go`.
- Проверить:
  - как сохраняются `RuleSets` и `rule` для пользовательских SRS;
  - как определяется тип `srs` (`DetermineRuleType` и константы).
- Добавить, если нужно, небольшое расширение модели для удобного доступа к `(tag, url)` при отрисовке Rules, **без изменения формата state** (например helper-методы на уровне бизнес-логики).

### 2.3 UI вкладки Rules

- Основной файл:
  - `ui/wizard/tabs/rules_tab.go`
- План:
  - найти существующую реализацию кнопки SRS для встроенных правил (из 004);
  - расширить построение списка правил так, чтобы:
    - пользовательские правила типа `srs` тоже получали кнопку SRS;
    - состояние кнопки определялось на основании `HasLocalSRS` для всех тегов правила;
    - клик `⬇/✔️` вызывал общий путь `DownloadSRSGroup`.
  - учесть сценарий авто-скачивания при первом открытии Rules:
    - хранение флага «инициализация уже выполнена в этой сессии»;
    - обход только включённых пользовательских SRS-правил.

### 2.4 Генерация конфига

- Файл:
  - `ui/wizard/business/create_config.go`
- Задачи:
  - локализовать место, где из шаблона + custom rules собирается секция `route.rule_set`;
  - внедрить функцию вида `BuildRuleSetConfig(tag string, remoteURL *string) (RuleSetConfig, error)`, которая:
    - проверяет наличие локального файла по tag;
    - возвращает конфиг с `type: "local"/"remote"`, `path` или `url` соответственно;
  - использовать её:
    - при генерации rule_set для встроенных SRS;
    - при генерации rule_set для пользовательских SRS (remote URL из state).

## 3. Этапы реализации

### Этап 1: Анализ и вынос общей логики

1. Найти существующую реализацию скачивания и кнопки SRS (004/005).
2. Вынести/обобщить функции работы с `{ExecDir}/bin/rule-sets` и проверкой локальных файлов.
3. Добавить единый helper для генерации rule_set (`local`/`remote`).

### Этап 2: Интеграция с пользовательскими SRS в моделях

1. Убедиться, что для пользовательских SRS-правил есть доступ к списку `(tag, url)` на уровне бизнес-логики.
2. При необходимости добавить helper-структуру/метод без изменения формата state.

### Этап 3: Обновление UI Rules tab

1. Расширить построение списка правил:
   - добавить кнопку SRS к пользовательским правилам типа `srs`;
   - подключить обобщённый механизм определения состояния и скачивания.
2. Реализовать авто-скачивание для включённых пользовательских SRS-правил при первом открытии Rules.

### Этап 4: Обновление генерации конфига

1. Подключить helper `BuildRuleSetConfig` в `create_config.go`.
2. Убедиться, что:
   - встроенные правила с SRS продолжают работать как в 004;
   - пользовательские SRS-правила автоматически используют `local` при наличии локального файла.

### Этап 5: Тесты и регрессии

1. Unit-тесты для helper-функций работы с путями и выбором `local`/`remote`.
2. Интеграционные тесты wizarda:
   - пользовательское правило SRS с одним и несколькими URL + локально скачанные файлы → в конфиге `type: "local"` и `path`.
3. Ручная проверка UI:
   - кнопка SRS для пользовательских правил;
   - авто-скачивание и поведение при ошибках.

## 4. Зависимости и риски

- Зависимость от задач:
  - `004-F-C-SRS_LOCAL_DOWNLOAD` — используется существующий стек скачивания и UI-паттерн;
  - `014-F-C-RULE_TYPE_SRS_URL` и `018-F-C-CUSTOM_RULE_SUBSYSTEM_REFACTOR` — модель пользовательских SRS-правил.
- Риски:
  - разная логика для встроенных и пользовательских правил может быть реализована сейчас в разных местах; важно не разнести функционал ещё сильнее, а наоборот **сконцентрировать** в общих helper-ах;
  - аккуратность с авто-скачиванием, чтобы не запускать множественные параллельные загрузки при повторном открытии Rules.
