# План: Отдельное окно логов (архитектура для согласования)

## 1. Общая идея

- На вкладке **Diagnostics** — кнопка «Open logs» (или «Logs»); по нажатию открывается **отдельное окно** (fyne.Window). В окне — **три вкладки**: **Internal**, **Core**, **API**.
- **Вкладка Internal**: источник — только перехват в `debuglog.Log()` (sink). На вкладке задаётся **уровень** (Error, Warn, Info, Verbose/Debug, Trace); в список попадают только события с уровнем не ниже выбранного. При открытии окна регистрируется sink через `debuglog.SetInternalLogSink(fn)`, при закрытии — `debuglog.ClearInternalLogSink()`. Sink не должен блокировать (push в channel, обновление UI через fyne.Async). Файл для Internal не используется.
- **Вкладка Core**: источник — только файл `logs/sing-box.log` (tail при открытии вкладки, далее автообновление раз в 5 секунд и по кнопке «Refresh» при активной вкладке). Фильтр по уровню не применяется; уровни в строках можно отображать визуально (парсинг). Макс. 300 строк в отображении.
- **Вкладка API**: источник — **перехват в месте генерации** (sink в `api.writeLog()`), как для Internal в debuglog и как для Core из файла. При открытии окна регистрируется sink через `api.SetAPILogSink(fn)`, при закрытии — `api.ClearAPILogSink()`. Sink вызывается с `(level, line)` для каждой записи; вкладка показывает события в реальном времени, опционально фильтр по уровню. Макс. 300 событий. Файл для отображения не читается.

## 2. Компоненты

| Компонент | Назначение |
|-----------|------------|
| **Чтение логов (Core)** | Tail по файлу `logs/sing-box.log`: прочитать последние N строк (макс. 300 в отображении). Internal и API из файла не читаются — собираются из места генерации (sink). В виджетах Internal и API хранить не более 300 последних событий. |
| **Перехват API** | В пакете `api`: при каждом `writeLog()` после записи в apiLogFile вызывать зарегистрированный sink `func(level debuglog.Level, line string)` если он установлен. Публичный API: `SetAPILogSink(fn)`, `ClearAPILogSink()`. Sink не должен блокировать. Аналогично перехвату Internal в debuglog. |
| **Парсинг уровня** | Internal и API: уровень приходит в sink. Core: по ключевым словам в строке (error, warn, info, debug, trace) для визуальной подсветки; фильтр по уровню для Core не применяется. |
| **Перехват internal** | В `internal/debuglog`: при каждом `Log()` после вывода в log вызывать зарегистрированный sink `func(level Level, line string)` если он установлен. Публичный API: `SetInternalLogSink(fn)`, `ClearInternalLogSink()`. Sink вызывается из любой горутины; окно по выбранному уровню показывает только подходящие записи Internal (level не ниже выбранного). Sink не должен блокировать. |
| **Фильтр по уровню** | На вкладках Internal и API (уровень приходит в sink). На вкладке Core фильтр по уровню не используется. |
| **UI** | Fyne: кнопка на вкладке Diagnostics открывает новое окно (fyne.Window). В окне — **три вкладки**: **Internal**, **Core**, **API**. На Internal и API: виджет логов из sink (в месте генерации), выбор уровня, макс. 300 событий. На Core: кнопка «Refresh», виджет из файла (tail), автообновление раз в 5 секунд при активной вкладке Core, макс. 300 строк. При открытии окна — регистрация sink в debuglog и в api; при закрытии — снятие обоих sink. Окно закрывается по кнопке или крестику. |

## 3. Поток данных

1. Пользователь на вкладке Diagnostics нажимает «Open logs» → открывается отдельное окно с тремя вкладками. При открытии вызываются `debuglog.SetInternalLogSink(callback)` и `api.SetAPILogSink(callback)` (sink активны пока окно открыто).
2. **Вкладка Internal**: выбор уровня; перехват в `debuglog.Log()` передаёт в callback `(level, line)`; отображаются только строки с уровнем не ниже выбранного. Визуальная индикация уровня.
3. **Вкладка API**: сбор из места генерации — перехват в `api.writeLog()` передаёт в callback `(level, line)`; отображаются события в реальном времени, опционально фильтр по уровню, макс. 300 событий. Визуальная индикация уровня.
4. **Вкладка Core**: при переключении на неё или по «Refresh» загружается tail из файла `logs/sing-box.log` (макс. 300 строк), далее при активной вкладке Core работает таймер автообновления раз в 5 секунд. Визуальная индикация уровня по парсингу строки. Таймер автообновления должен удаляться при закрытии окна логов или при переключении с вкладки Core.
5. При закрытии окна вызываются `debuglog.ClearInternalLogSink()` и `api.ClearAPILogSink()`.

## 4. Размещение кода

- **Tail-чтение (только для Core)**: функция в `internal/` или `core/services/file_service.go`: `ReadLastLines(path string, maxLines int) ([]string, error)`. Использовать только для `logs/sing-box.log` (Core). Internal и API из файла не читаются — собираются из места генерации (sink).
- **Парсинг уровня**: для Core — по ключевым словам в строке (для подсветки); для Internal и API уровень приходит из sink.
- **debuglog**: экспорт `SetInternalLogSink(fn func(Level, string))` и `ClearInternalLogSink()`. В `Log()` после записи в log вызывать sink под RLock, если установлен. Sink предназначен для окна логов; callback не должен блокировать.
- **api**: экспорт `SetAPILogSink(fn func(debuglog.Level, string))` и `ClearAPILogSink()`. В `writeLog()` после записи в apiLogFile вызывать sink (то же формирование строки, что пишется в файл). Sink не должен блокировать. Сбор API-логов из места генерации.
- **UI**: в `ui/diagnostics_tab.go` — кнопка «Open logs»; по нажатию создаётся отдельное окно (логи можно вынести в `ui/log_viewer_window.go`). В окне — **три вкладки**: **Internal** и **API** — виджеты из sink (место генерации), выбор уровня, макс. 300 событий; **Core** — Refresh + виджет из файла `logs/sing-box.log`. При показе окна — SetInternalLogSink и SetAPILogSink; при закрытии — ClearInternalLogSink и ClearAPILogSink. Путь к файлу Core — через `AppController.FileService.ExecDir` и `constants.ChildLogFileName`.

## 5. Зафиксированные решения

| Пункт | Решение | Примечания |
|-------|---------|-----------|
| **Обновление Core** | Кнопка «Refresh» + автообновление по таймеру раз в 5 секунд при активной вкладке Core. | Таймер автообновления удаляется при закрытии окна логов или при переключении на другую вкладку. |
| **Структура окна** | Три вкладки: Internal, Core, API. Internal и API — сбор из места генерации (sink в `debuglog.Log()` и `api.writeLog()`); Core — из файла (tail). | — |
| **Лимит** | Макс. 300 строк (событий) в отображении на каждой вкладке. При tail для Core читать из файла не более 300 последних строк. | При необходимости лимит можно скорректировать отдельно. |
| **Путь к файлу Core** | Строить путь **только через ту же переменную**, что и при запуске sing-box: `filepath.Join(ac.FileService.ExecDir, childLogFileName)` (в `core/controller.go`: `childLogFileName = "logs/" + constants.ChildLogFileName`). Дублировать конкатенацию «logs» + имя файла в других местах нельзя — изменение только в одном месте. | — |

## 6. Риски и ограничения

- Файлы логов могут быть открыты на запись (MainLogFile, ChildLogFile, ApiLogFile); чтение на Windows/других ОС обычно возможно одновременно с записью. Если окажется иначе — учесть копирование в буфер или предупреждение.
- Большие файлы: не читать целиком, только tail (seek с конца, читать блоками до набора N строк или N байт).
