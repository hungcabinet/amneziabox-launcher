# Спецификация: Окно логов (Diagnostics)

## 1. Проблема

На вкладке **Diagnostics** сейчас есть только инструменты проверки IP (STUN, ссылки на 2ip, Yandex и т.д.). Пользователь не может посмотреть логи приложения и ядра (sing-box) без открытия файлов вручную.

## 2. Цель

Добавить **отдельное окно просмотра логов**, открываемое с вкладки Diagnostics (кнопка «Open logs» / «Logs»). В окне — **три вкладки**: **Internal**, **Core**, **API**.

- **Вкладка Internal**: логи лаунчера из перехвата `debuglog.Log()` (sink). На вкладке задаётся **уровень** (Error | Warn | Info | Verbose/Debug | Trace); в список попадают только события с уровнем не ниже выбранного (перехват передаёт каждую запись в sink `(level, line)`, окно отображает только подходящие).
- **Вкладка Core**: логи sing-box из файла `logs/sing-box.log` (tail при открытии вкладки, далее автообновление раз в 5 секунд и по кнопке «Refresh» при открытом окне). Фильтр по уровню не применяется; уровни в строках можно отображать визуально (парсинг).
- **Вкладка API**: логи API-запросов (Clash API и т.п.) **собираются из места генерации** — перехват в `api.writeLog()` (sink), как для Internal в `debuglog.Log()` и как для Core из файла (Core генерируется процессом sing-box в файл, мы читаем файл; API генерируется в пакете `api` в `writeLog` — добавляем sink там). В реальном времени при открытом окне записи дублируются в вкладку API; опционально фильтр по уровню (в `writeLog` уже передаётся уровень). Макс. 300 событий в отображении.

## 3. Требования (кратко)

- На вкладке Diagnostics — кнопка, по нажатию открывается **отдельное окно** с просмотром логов (read-only).
- В окне логов — **три вкладки**:
  - **Internal**: выбор **уровня** — Error | Warn | Info | Verbose (Debug) | Trace (как в `debuglog.Level`). В список дублируются только события, попадающие под выбранный уровень (перехват в `debuglog.Log()` → sink `(level, line)`; окно решает, показывать ли строку). Визуальная индикация уровня каждой записи (цвет/иконка/метка).
  - **Core**: содержимое из файла `logs/sing-box.log` (tail), кнопка «Refresh», автообновление раз в 5 секунд при активной вкладке Core, визуальная индикация уровня по парсингу строки; фильтр по уровню не используется.
  - **API**: перехват в месте генерации (`api.writeLog()` → sink), в реальном времени; опционально фильтр по уровню; макс. 300 событий в отображении.
- Источники данных:
  - **Internal**: только перехват в `debuglog.Log()` (sink), в реальном времени при открытом окне. Формат строк — префиксы `[DEBUG]`, `[INFO]`, `[WARN]`, `[ERROR]`; уровень передаётся в sink.
  - **Core**: только файл `logs/sing-box.log` (место, куда пишет sing-box); объём — последние N строк (tail, макс. 300 в отображении), обновление по кнопке «Refresh» и автообновление по таймеру раз в 5 секунд при активной вкладке Core. Таймер должен удаляться при закрытии окна логов или при переключении с вкладки Core.
  - **API**: только перехват в `api.writeLog()` (sink в месте генерации), в реальном времени при открытом окне. Уровень передаётся в sink; вкладка может фильтровать по уровню. Файл для отображения не читается (как и для Internal).
- Тексты в UI — только английский (constitution).

## 4. Критерии приёмки

- Пользователь открывает окно логов с вкладки Diagnostics. На вкладке **Internal** задаёт уровень — отображаются только события из перехвата `debuglog.Log()`, проходящие фильтр. На вкладке **API** — события из перехвата `api.writeLog()` (место генерации), опционально с фильтром по уровню; макс. 300 событий. На вкладке **Core** показывается содержимое файла (tail, макс. 300 строк).
- Уровень критичности каждой записи виден (цвет, иконка или подпись).
- При отсутствии файла лога или ошибке чтения — корректное сообщение (например, «Log file not available»), без падения приложения.
- Сборка и тесты проходят; логирование через debuglog в новых путях кода.

## 5. Архитектурные решения

- Чтение логов для вкладки Core: только из файла (`logs/sing-box.log`) через tail без дополнительного буфера для «текущей сессии».
- Обновление вкладки Core: по кнопке «Refresh» и автообновление по таймеру раз в 5 секунд при активной вкладке Core; таймер удаляется при закрытии окна логов или при переключении на другую вкладку.
- Ограничение: до 300 строк (событий) в отображении на каждой вкладке.
