# Спецификация: лаунчер «Не отвечает» после сна/гибернации (Windows)

Проблема и исправление касаются **только Windows**; на macOS и Linux не воспроизводится и отдельная обработка не требуется.

## Проблема

После выхода Windows из сна или гибернации окно лаунчера перестаёт реагировать: в заголовке и в диспетчере задач отображается «Не отвечает». Процесс sing-box при этом продолжает работать (есть CPU/сеть). Через несколько минут (наблюдалось ~7) лаунчер сам восстанавливается.

### Симптомы

- Окно лаунчера: «Не отвечает» в заголовке; в диспетчере — «VPN singbox-launcher» / «Singbox Launcher» с пометкой «Не отвечает».
- sing-box: работает, процесс не помечен как не отвечающий.
- Зависший лаунчер: 0% CPU, нет диска/сети — типичная блокировка главного (UI) потока.
- Через ~7 минут лаунчер снова отвечает без перезапуска.

### Воспроизведение

1. Запустить лаунчер, при необходимости запустить VPN (sing-box).
2. Открыть вкладку Servers (Clash API On, список прокси с задержками).
3. Перевести Windows в сон или гибернацию.
4. Разбудить систему.
5. Лаунчер показывает «Не отвечает»; через несколько минут (наблюдалось ~7) окно снова отвечает.

Окружение: Windows 10/11.

**Наблюдение:** старая версия часто не зависает при короткой гибернации (минуты), но стабильно зависает при гибернации на часы. Вероятные причины: (1) при коротком сне TCP-соединения к Clash API ещё могут быть в допустимом состоянии или быстро обрываются; при долгом сне соединения гарантированно мёртвые, и попытка использовать их из пула приводит к долгому ожиданию (таймауты TCP/драйвера) и/или срабатыванию проблемного пути в UI. (2) После долгого сна сильнее инвалидируется состояние GPU/окна, и следующий рендер или системный вызов с большей вероятностью блокируется. Исправление (сброс HTTP-транспорта при resume, IdleConnTimeout, отмена запросов при sleep) устраняет повторное использование «протухших» соединений и снижает риск блокировки.

### Гипотезы

1. **Главный поток блокируется в системном вызове после resume**  
   Драйвер (OpenGL/GPU) или очередь сообщений Windows после выхода из сна попадает в состояние, при котором следующий вызов (рендер, сообщение) «залипает» и возвращается только по внутреннему таймауту ОС/драйвера (~5–7 минут). Объясняет и зависание, и самовосстановление через фиксированное время.

2. **Нет обработки событий питания**  
   Приложение не подписывается на WM_POWERBROADCAST (PBT_APMRESUMESUSPEND / PBT_APMRESUMEAUTOMATIC) и не реагирует на пробуждение: не сбрасывает HTTP-транспорт к Clash API. «Протухшие» TCP-соединения после сна могут усиливать нагрузку или порядок вызовов; блокировка на ~7 минут скорее указывает на блокировку не в нашем коде.

3. **Глобальный HTTP-клиент (api/clash.go)**  
   Один `http.Client` без IdleConnTimeout; после сна соединения из пула могут быть полуживыми. Запросы к API идут в горутинах (таймаут 20 с), сами по себе не блокируют UI, но при большом числе операций после пробуждения возможна дополнительная нагрузка.

4. **Deadlock (маловероятно для ~7 мин)**  
   Главный поток ждёт мьютекс, горутина держит мьютекс и ждёт HTTP (макс. 20 с). 7 минут не укладываются в один такой цикл; менее правдоподобно, чем долгий таймаут в драйвере/ОС.

## Цель

После выхода системы из сна/гибернации лаунчер либо не зависает, либо восстанавливается быстрее. Минимум: снизить риск и последствия «протухших» соединений и отсутствия реакции на resume (обработка событий питания, сброс HTTP-транспорта).

## Требования

1. **Обработка выхода из сна (Windows)** — подписка на power/session события (WM_POWERBROADCAST или аналог в используемом стеке). При resume: сброс HTTP-транспорта к Clash API (пересоздание клиента или транспорта); при необходимости — лёгкое обновление UI (без тяжёлых операций на главном потоке).
2. **HTTP-транспорт (api/clash.go)** — задать IdleConnTimeout в Transport, чтобы не переиспользовать долгоживущие соединения после длительного простоя/сна.
3. **Исследование** — по возможности проверить issues Fyne на тему Windows sleep/hibernate и OpenGL; при наличии известных проблем рассмотреть обход (пересоздание окна/контекста после resume, если API позволяет).

Реализация — в рамках constitution и IMPLEMENTATION_PROMPT (логирование через debuglog, ошибки пользователю через dialogs, минимальный дифф).

## Архитектура: события питания (platform)

Платформа (**internal/platform**) даёт единый контракт для реакции на сон/пробуждение. **Клиенты не зависят от ОС:** они всегда вызывают одни и те же API (IsSleeping, PowerContext, RegisterPowerResumeCallback, StopPowerResumeListener); платформа сама решает, слать события и выставлять статус или нет (на Windows — полная реализация, на остальных ОС — no-op и константный статус).

1. **Два события для подписки**
   - **Sleep** — система уходит в сон. Подписчики должны прервать текущую работу (сетевые запросы — обрыв, таймеры — пропуск действия) и не начинать новую, пока не наступит resume.
   - **Resume** — система вышла из сна/гибернации. Подписчики могут возобновить работу; при необходимости платформа инициирует сброс ресурсов (например, HTTP-транспорт).

2. **Текущий статус sleep**
   - Читаемый флаг: `IsSleeping() == true` между событиями sleep и resume. Все, кто планирует действие (HTTP-запрос, таймер), проверяют этот статус через platform; при sleep не начинают запрос и скипают действие.

Подписчики (api, таймер меню трея, AutoLoadProxies, UI) используют проверку статуса и/или подписку на события; платформа инкапсулирует детали ОС.

## Критерии приёмки

- [ ] На Windows при выходе из сна/гибернации выполняется обработка события resume (подписка на power/session события).
- [ ] При resume сбрасывается HTTP-транспорт к Clash API (новый клиент или новый Transport), чтобы не использовать «протухшие» соединения.
- [ ] В `api/clash.go` у Transport задан IdleConnTimeout (конкретное значение — в PLAN).
- [ ] Сборка и тесты проходят; линтер без новых ошибок. В новых путях кода — debuglog при start/success/error.
- [ ] (Желательно) После resume лаунчер не показывает «Не отвечает» или восстанавливается заметно быстрее; при невозможности устранить блокировку в драйвере — документировать в IMPLEMENTATION_REPORT ограничения.
