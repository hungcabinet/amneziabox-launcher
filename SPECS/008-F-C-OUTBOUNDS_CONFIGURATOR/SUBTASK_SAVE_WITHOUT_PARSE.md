# Подзадача: сохранение без парсинга outbounds

**Фича:** OUTBOUNDS_CONFIGURATOR  
**Приоритет:** опционально (улучшение UX и независимость от сети)

---

## Цель

Сделать кнопку **Save** в визарде независимой от наличия соединения: не запускать парсинг/генерацию outbounds при сохранении, а сохранять текущее состояние конфига; после успешного сохранения при необходимости запускать процесс обновления конфига с главной вкладки (Update).

## Текущее поведение

- При нажатии Save вызывается `SaveConfig()` → `executeSaveOperation()` → **`waitForParsingIfNeeded()`**.
- Если нужен парсинг (`PreviewNeedsParse` или уже идёт `AutoParseInProgress`), сохранение **ждёт** завершения `ParseAndPreview` (загрузка подписок по URL + `GenerateOutboundsFromParserConfig`).
- Без сети или при медленных подписках Save может долго висеть или упасть по таймауту.

## Предлагаемое поведение

1. **При Save** не вызывать парсинг outbounds:
   - Сразу строить конфиг из текущей модели (`BuildTemplateConfig`) — те `GeneratedOutbounds`, что уже есть (или пусто/дефолт, если парсинг ещё не запускали).
   - Сохранять файл, сохранять state, показывать диалог успеха.
   - Убрать или существенно упростить `waitForParsingIfNeeded()` в потоке Save (не ждать парсинга для записи на диск).

2. **После успешного Save** (опционально):
   - Запускать процесс «Update» с главной вкладки (обновление конфига из подписок), если такой механизм есть в приложении.
   - Либо оставить обновление на усмотрение пользователя (кнопка Update на главной).

## Что проверить в коде

- **Где вызывается Save:** `ui/wizard/presentation/presenter_save.go` — `SaveConfig`, `executeSaveOperation`, `waitForParsingIfNeeded`.
- **Откуда берётся конфиг при Save:** `buildConfigForSave()` → `BuildTemplateConfig(model, …)`; в модели уже лежат `model.GeneratedOutbounds` из последнего парсинга.
- **Где живёт «Update с главной»:** поиск в `core/` и главном окне — кнопка/действие обновления конфига из подписок; при необходимости вызвать его после закрытия визарда по Save (или передать флаг «только что сохранили, запустите update»).

## Чеклист (при реализации)

- [x] Убрать ожидание парсинга в потоке Save: не вызывать `waitForParsingIfNeeded()` в `executeSaveOperation()` или вызывать только когда без парсинга конфиг нельзя собрать (уточнить по модели).
- [x] Убедиться, что `buildConfigForSave()` корректно работает с уже имеющимися `GeneratedOutbounds` (и с пустым состоянием, если парсинг ни разу не запускали).
- [x] Найти точку входа «Update config from subscriptions» на главной и при необходимости вызвать её после успешного Save (или задокументировать, что пользователь может нажать Update вручную).
- [x] Обновить статусы/подписи в UI (убрать «Updating outbounds...» из Save, если парсинг при Save больше не выполняется).
- [ ] Проверить сценарии: Save без сети, Save с устаревшим превью, первый Save без предварительного парсинга.

---

## Реализовано

- В `executeSaveOperation()` убрано ожидание парсинга: вызов `waitForParsingIfNeeded()` и сама функция удалены. Save сразу строит конфиг из текущей модели (`buildConfigForSave()` → `BuildTemplateConfig`) и пишет файл.
- После успешного сохранения в `completeSaveOperation()` вызывается `go core.RunParserProcess()` — тот же процесс, что и кнопка Update на главной вкладке (обновление конфига из подписок в фоне).
- Статус «Updating outbounds...» больше не показывается при Save (логика удалена вместе с `waitForParsingIfNeeded`).
- Комментарии в `presenter_save.go` обновлены под новое поведение.
