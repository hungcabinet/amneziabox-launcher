# Constitution — принципы проекта singbox-launcher

Неизменяемые принципы и ограничения, которым должны соответствовать все решения в проекте.

---

## 1. Приоритеты

**Главный приоритет — стабильность.** Надёжная работа важнее новых фич. Перед релизом — проверка на регрессии, graceful degradation при сбоях.

---

## 2. Архитектура

### 1.1 Разделение ответственности
- **Бизнес-логика** отделена от UI
- **Сервисы** инкапсулируют специфическую функциональность
- **Модели данных** отделены от обработки

### 1.2 Модульность
- Каждый функциональный блок — отдельный пакет
- Минимальные зависимости между модулями
- Подпакеты группируют связанную функциональность

### 1.3 Dependency Injection
- Сервисы получают зависимости через конструкторы
- Callback-функции для обратной связи
- Минимизация циклических зависимостей

### 1.4 Единая точка входа
- `AppController` координирует все компоненты
- Сервисы делегируют задачи, не вызывают друг друга напрямую без необходимости

### 1.5 Архитектурные инварианты (нельзя нарушать)
- **UI не обращается к core/network напрямую** — слой UI не обращается к `core/` или сетевым вызовам напрямую; только через контроллер/сервисы.
- **Парсер детерминирован и без побочных эффектов** — без сети, без записи, без глобального состояния.
- **Телеметрия не блокирует основной поток** — только асинхронно/в фоне.
- **Платформозависимый код изолирован в internal/platform** — условная компиляция по `runtime.GOOS` или build tags.

---

## 3. Платформа и технологии

### 2.1 Кроссплатформенность
- Поддержка **Windows**, **macOS**, **Linux**
- Платформозависимый код — в `internal/platform/`
- Один кодовая база, условная компиляция по `runtime.GOOS`

### 2.2 GUI
- **Fyne** — единственный UI-фреймворк
- Без зависимостей от нативных виджетов ОС

### 2.3 Ядро
- **sing-box** — внешний исполняемый файл, не библиотека
- Конфигурация — JSON/JSONC, совместимый с sing-box

---

## 4. Файловая структура

### 3.1 Директории
- `bin/` — исполняемые файлы (sing-box), шаблоны (wizard_template.json), конфиги
- `logs/` — лог-файлы приложения
- Пути строятся от `ExecDir` (директория исполняемого файла лаунчера)

### 3.2 Конфигурация
- `config.json` — основной конфиг (путь платформозависимый, см. `platform.GetConfigPath`)
- Шаблоны и ассеты — в `bin/`

---

## 5. Логирование

- **Централизованное** — пакет `internal/debuglog`
- Уровни: Debug, Info, Warn, Error
- Не использовать `log` напрямую для прикладной логики
- Логи — в файлы, не в stdout (кроме отладочного режима)
- **Новые участки кода** — обязательно `debuglog.DebugLog(...)` в ключевых точках: **start** операции, **success**, **error** (критично для диагностики в production)

---

## 6. Пользователи и регионы

### 6.1 Целевая аудитория
- **Регионы:** РФ, СНГ, глобально
- **Уровень пользователей:** от новичков до продвинутых
- UI и сообщения — понятны разным уровням; продвинутые настройки доступны, но не навязываются

### 6.2 Учёт ограничений доступа
- Проект ориентирован на пользователей в регионах с ограничениями (в т.ч. РФ)
- Не допускать зависимостей от заблокированных доменов при **первом запуске**
- Резервные источники (mirrors) для загрузок, когда основной недоступен

### 6.3 Приватность и данные (жёсткий запрет)
- ❌ **Деанонимизация** — нельзя собирать или комбинировать данные так, чтобы идентифицировать пользователя
- ❌ **Неявный сбор** — нельзя собирать данные без явного согласия (opt-in) и прозрачного объяснения
- ❌ Содержимое конфигов, домены, IP, идентификаторы устройств
- ✅ Allowlist: только явно разрешённые данные, только при opt-in

---

## 7. Спецификации и разработка

### 7.1 Spec-driven
- Крупные фичи и баги — в `SPECS/` (папки `NNN-T-S-NAME`, см. SPECS/README.md)
- Структура: `SPEC.md`, `PLAN.md`, `TASKS.md`, `IMPLEMENTATION_REPORT.md`
- SPEC — что и зачем; PLAN — как строить; TASKS — конкретные задачи

### 7.2 Обратная совместимость
- Миграции конфигов и состояний — с сохранением работоспособности старых версий
- Явная версионность форматов (например, `WizardStateFile`)

---

## 8. Качество кода

### 8.1 Тестирование
- **Unit-тесты** — для бизнес-логики (парсеры, генераторы конфигов, утилиты)
- **GUI-пакеты** (fyne) исключаются из `go test` — требуют OpenGL, запускать через скрипты в `build/`
- **Интеграционные тесты** — через `build/test_*.sh` с явным исключением GUI-пакетов
- Перед релизом — проверка на регрессии, ручное тестирование ключевых сценариев

### 8.2 Обработка ошибок
- **Единая точка** отображения ошибок пользователю — `dialogs.ShowError`, `ShowErrorText`
- **Сообщения** (все тексты в UI, диалогах, уведомлениях).
- Язык технический сообщений только английский
- Сообщения — понятные пользователю, без технического жаргона где возможно; контекст для отладки — в логах.
- **Логирование** — через `debuglog` (ErrorLog, WarnLog, InfoLog, DebugLog) с контекстом.
- **Не игнорировать** ошибки: `_ = err` допустимо только при явном обосновании в комментарии.

### 8.3 Стиль и читаемость
- **Язык интерфейса приложения** — английский.
- **Комментарии в коде** — на английском; длинные пояснения допустимы на русском.
- Имена переменных, функций, пакетов — на **английском**
- Сложная логика — выносить в отдельные функции с понятными именами
- Длинные функции — разбивать; один уровень абстракции на блок

### 8.4 Документация
- Пакеты — краткое описание в начале файла (package comment)
- Публичные функции — docstring с назначением, параметрами, возвращаемым значением
- Спецификации фич и багов — в `SPECS/NNN-T-S-NAME/` (SPEC, PLAN, TASKS)
- Языки документации русский и английский 

### 8.5 Implementation Prompt
- При реализации фич следовать [SPECS/IMPLEMENTATION_PROMPT.md](IMPLEMENTATION_PROMPT.md) — философия разработки, требования к коду, Definition of Done

## 10. Производительность

- ❌ **Блокирующие операции в UI-потоке** — сеть, диск, тяжёлые вычисления в главном потоке; выносить в горутины, обновлять UI через `fyne.Do` / существующие обёртки.
- ❌ **Лишние аллокации в горячих путях** — циклы, обработчики событий; переиспользовать буферы/слайсы где возможно.
- ❌ **Синхронные сетевые вызовы в ядре** — использовать таймауты, контекст и по возможности асинхронный подход.

---

## 11. Запреты

- ❌ Хардкод секретов, токенов, API-ключей
- ❌ Прямые вызовы UI из бизнес-логики (только через callbacks/presenter)
- ❌ Игнорирование ошибок (`_ = err`)
- ❌ Зависимость первого запуска от заблокированных доменов
- ❌ Деанонимизация пользователей или неявный сбор данных
