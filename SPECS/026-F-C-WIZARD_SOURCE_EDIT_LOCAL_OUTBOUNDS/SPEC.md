# SPEC: расширенный редактор источника (первая вкладка визарда) и исключение нод из глобальных outbounds

**Статус:** C (закрыта). **Тип:** F (feature).

---

## Проблема

1. **Первая вкладка (Sources):** настройки источника размазаны по списку и окну **View** без единого места для локальных **auto**/**select** и связанных полей `ParserConfig`.
2. **Парсер:** все ноды из `proxies[]` всегда участвуют в выборке для **глобальных** `ParserConfig.outbounds`. Нужно уметь исключать источник из этой выборки и при необходимости публиковать в глобальных группах только теги локальных **auto**/**select**.

## Цель

1. **View → Edit**, внутри подвкладки: **Настройки** (префикс, локальные auto/select, новые поля из раздела ниже, предупреждения), **Просмотр** (локальные outbounds источника + список нод как View), **JSON** (только чтение, `proxies[i]`).
2. Два новых необязательных поля на каждом элементе **`proxies[]`** — см. **«Новые поля `ProxySource`»**.
3. Обновить **`docs/ParserConfig.md`** (и при необходимости шаблоны) — в **PLAN.md** перечень правок документации.

## Новые поля `ProxySource` (`ParserConfig.proxies[]`)

Оба поля — `bool`, в JSON с **`omitempty`**, по умолчанию как у отсутствующего ключа. **Логика друг от друга не зависит** (отдельные чекбоксы и отдельные эффекты).

| JSON-поле | Чекбокс в UI (RU, ориентир) | Эффект |
|-----------|-----------------------------|--------|
| **`exclude_from_global`** | Исключить из глобальных списков | При **`true`** ноды этого источника **не попадают** в пул кандидатов (`filteredNodes` / эквивалент) для **каждого** элемента **`ParserConfig.outbounds`** — обход **всего** массива, тип записи (**`selector`**, **`urltest`**, …) **не** ограничивать (**аналогично** **`expose_group_tags_to_global`** по охвату). Локальные `proxies[i].outbounds` по-прежнему используют только ноды этого источника. Распространяется на все ноды источника, включая WireGuard в **`endpoints`**. |
| **`expose_group_tags_to_global`** | Теги в глобальных группах | При **`true`** при **формировании** глобальных outbounds (генерация / превью — **PLAN §3**) теги локальных групп источника (по §1–§2: только auto, только select или оба) **кандидаты** на добавление в эффективный список для **каждого** элемента **`ParserConfig.outbounds`**. Если у записи заданы **`filters`** — кандидат добавляется **только** при прохождении того же правила, что и для нод (**§5**). В **сохранённом JSON** `ParserConfig.outbounds[].addOutbounds` эта галочка **ничего не записывает и не удаляет**; строки из **`addOutbounds`** по-прежнему **не** прогоняются через **`filters`** (как в текущем генераторе — **§5**). При **`false`** expose-теги **не** подмешиваются. |

**Синхронизация при смене локальных групп:** если не осталось ни auto, ни select — выставить **`expose_group_tags_to_global`** в **`false`** (в **`proxies[i]`**); если исчезла одна из двух групп — флаг может остаться **`true`**, генератор подмешивает только оставшийся тег.

## Парсер и остальная модель

1. **`exclude_from_global`:** см. таблицу; реализация — на этапе **формирования** глобальных outbounds (**`buildOutboundsInfo`** / генератор): фильтрация пула нод для **всех** глобальных **`ParserConfig.outbounds`**. В JSON глобальных outbounds флаг **не** дописывает и не стирает поля — только логика кода (**PLAN.md**).
2. **`ParsedNode.SourceIndex`** (или эквивалент): сопоставление ноды с индексом `proxies[i]` при генерации; без индекса — не исключать из глобальной выборки (или всегда выставлять в генераторе — **PLAN.md**).
3. **Версия ParserConfig:** по возможности **4** + optional поля; иначе миграция — **PLAN.md**.
4. Локальные **auto/select** в JSON — как обычные элементы **`proxies[i].outbounds`**, формат согласовать с **ParserConfig.md** и **`GenerateSelectorWithFilteredAddOutbounds`**.

## Риски

- Коллизии тегов — префикс **`tag_prefix`**; логи парсера при дублях.
- **`expose`:** теги попадают в **каждую** глобальную группу на этапе генерации — осознанный охват; при необходимости дедуп с тегами, уже перечисленными в JSON **`addOutbounds`**.
- Все источники с exclude и без **`expose`** / явных ссылок в пользовательском **`addOutbounds`** — риск пустых глобальных списков; предупреждение в UI снижает риск.

## Вне объёма

- Персистентная мутация **`ParserConfig.outbounds[].addOutbounds`** ради **`expose_group_tags_to_global`** / **`exclude_from_global`** (оба — только поведение генератора).
- Смена формата подписок, кроме **`SourceIndex`** у нод.

## Принятые решения (для реализации)

Детали кода — **PLAN.md**.

### §1. Теги локальных групп

| Роль | Тег | Пример при `tag_prefix` = `1:` |
|------|-----|--------------------------------|
| auto | `trim(tag_prefix) + "auto"` | `1:auto` |
| select | `trim(tag_prefix) + "select"` | `1:select` |

Пустой префикс — назначить как при **Add** (`GenerateTagPrefix`). Смена префикса — переименовать теги у записей с маркерами §2 и обновить select → auto.

### §2. Маркеры в `comment` (`proxies[i].outbounds`)

| Галочка | Подстрока в `comment` |
|---------|------------------------|
| auto | `WIZARD:auto` |
| select | `WIZARD:select` / `WIZARD:selector` (достаточно искать `WIZARD:select`) |

Вкл галочки при отсутствии записи — шаблон с маркером. Выкл — удалить все записи с этим маркером у данного `i`. Конфликт тега с записью **без** маркера — предупреждение, не перезаписывать. Пользователь может править поля записей; визард не затирает при открытии. Для предсказуемой фильтрации expose по полю **`comment`** в **`ParserConfig.outbounds[].filters`** маркированные записи должны **стабильно содержать** указанные подстроки (**§5**); визард при создании/шаблоне гарантирует это, ручное редактирование — ответственность пользователя.

### §3. Локальный `selector`

Пул — `nodesBySource[i]`; **без фильтра** или **`{}`** — все ноды пула. Тег **auto** в **`addOutbounds`**; порядок с нодами — **`GenerateSelectorWithFilteredAddOutbounds`**.

### §4. Префикс в списке источников

Редактирование только в **Настройках** Edit; в строке списка — только показ префикса.

### §5. `expose_group_tags_to_global` и `ParserConfig.outbounds[].filters`

1. **Строки из JSON `addOutbounds`** для глобального outbound по-прежнему **не** проходят через **`filters`** (поведение как в **`GenerateSelectorWithFilteredAddOutbounds`** сейчас: фильтр только отбирает **ноды** из `allNodes`). Меняется только судьба **подмешиваемых** expose-тегов.

2. **Expose-кандидат** (тег локальной группы, напр. `1:auto`) перед добавлением в эффективный список для данной записи **`ParserConfig.outbounds[]`** проверяется тем же механизмом, что и нода: **`matchesFilter`** / OR-массив **`filters`**, AND внутри объекта — см. **`outbound_filter.go`**. Для проверки строится **синтетический** набор полей, совместимый с **`getNodeValue`**:
   - **`tag`** — тег группы (§1);
   - **`comment`** — значение **`comment`** у соответствующей записи **`proxies[i].outbounds`** с маркером §2 (чтобы фильтры вроде `"/WIZARD:auto/i"` по ключу **`comment`** работали предсказуемо);
   - **`host`**, **`scheme`**, **`label`**, **`fragment`** — **пустые строки** (если позже не зафиксировано иное в **IMPLEMENTATION_REPORT**).

3. **AND с полями вроде `host`:** при пустом **`host`** у синтетики условие по **`host`** **не выполняется** → expose-тег в этот глобальный outbound **не** попадает. Это ожидаемо: пользователь сузил группу до «реальных» нод с конкретным host.

4. **`filters` отсутствует или пустой `{}`** — как для нод: ограничения нет, все подходящие expose-кандидаты подмешиваются (с учётом дедупа и **`exclude_from_global`** для нод — отдельно).
