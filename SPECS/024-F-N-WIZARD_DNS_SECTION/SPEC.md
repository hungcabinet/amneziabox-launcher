# SPEC: WIZARD_DNS_SECTION — вкладка DNS в визарде

**Статус:** N (новая). **Тип:** F (feature).

**Объём этого артефакта:** только SPEC.md; PLAN.md / TASKS.md — отдельно, после утверждения требований.

---

## Проблема

Секция `config.dns` и связанное поле `config.route.default_domain_resolver` задаются только через `wizard_template.json` (и при сборке конфига копируются в выходной JSON как есть, без участия модели визарда). Пользователь не может в UI:

- увидеть и изменить список DNS-серверов;
- выбрать `dns.final` и резолвер по умолчанию для маршрутизации;
- править `dns.rules` без ручного редактирования шаблона или готового `config.json`.

## Цель

Добавить во визард отдельную вкладку **DNS**, на которой редактируются данные, попадающие в итоговый `config.json` в объекте `dns` и в `route.default_domain_resolver`, с сохранением совместимости с форматом sing-box и с текущим шаблоном как **начальным состоянием**.

Источник начальных значений — эффективный `config` шаблона (с учётом `GetEffectiveConfig` на macOS при наличии `params`, как для остальных секций).

## Область данных (sing-box)

### Объект `dns` (визард обязан поддерживать как минимум)

| Поле | Поведение во визарде |
|------|----------------------|
| `servers` | Список серверов: показать все из шаблона; **добавление** и **удаление** записей; редактирование полей сервера (тип `local` / `udp` / `tcp` / `tls` / `https` / `h3` / `quic` / `dhcp` / `fakeip` и т.д. — в объёме, достаточном для типовых записей из `bin/wizard_template.json` и документации sing-box). У каждого сервера обязателен стабильный **`tag`** (уникальный в списке). |
| `rules` | Редактирование в **простом текстовом формате** (см. ниже); при сохранении/превью парсинг в массив объектов правил sing-box. |
| `final` | Выпадающий список (или аналог): значение — **один из `tag`** текущего списка `dns.servers`. Должна быть возможность выбрать любой существующий тег; при удалении сервера с выбранным `final` — валидировать и требовать смены `final` или подставлять безопасное значение по правилам реализации (зафиксировать в PLAN). |
| `strategy` | Если присутствует в шаблоне — отобразить и дать менять (значения sing-box: например `prefer_ipv4`, `prefer_ipv6`, `ipv4_only`, `ipv6_only`). |
| `independent_cache` | Если присутствует в шаблоне — отобразить (bool). |

Прочие поля объекта `dns` из шаблона (если появятся) желательно сохранять через round-trip: не терять неизвестные ключи при сериализации, либо явно перечислить поддерживаемые ключи в PLAN.

### Поле `route.default_domain_resolver`

- Отдельный элемент UI на вкладке DNS (или логически связанный блок): выбор **одного тега** из `dns.servers` либо пустое значение «не задано» (если sing-box допускает отсутствие поля — оставлять ключ опущенным или как в шаблоне по умолчанию).
- Значение записывается в **`config.route`**, не внутрь `dns`.

При генерации конфига порядок: правки DNS и `default_domain_resolver` должны попадать в итоговый JSON так же, как сейчас мержится `route` (отдельная логика от «простого» копирования секции шаблона), чтобы вкладка DNS реально влияла на preview и save.

## Формат текста для `dns.rules` (требование «простой текст»)

Редактор — многострочное поле. Одна строка — **одно правило**.

**Синтаксис строки (MVP):** компактный JSON-объект в одну строку (NDJSON-подход), эквивалентный одному элементу массива `dns.rules` в sing-box.

Примеры (как в шаблоне):

```text
{"rule_set":"ru-domains","server":"yandex_doh"}
{"server":"google_doh"}
```

Пустые строки и строки, начинающиеся с `#` после optional whitespace — комментарии, игнорируются при разборе.

**Валидация:** при невалидном JSON в строке — показать понятную ошибку пользователю (язык UI по CONSTITUTION); не сохранять конфиг с некорректными правилами до исправления или отката.

**Расширение (опционально, не блокер MVP):** в PLAN можно заложить второй режим «ключ=значение» для самых частых полей; в SPEC достаточно NDJSON-строк как однозначного и простого для копирования из документации sing-box формата.

## Поведение для пользователя (кратко)

1. Открыть вкладку **DNS**.
2. Видеть список всех DNS-серверов, как в шаблоне (тег, тип, краткая сводка адреса/пути).
3. **Добавить** сервер (диалог или форма по аналогии с другими редакторами визарда), **удалить** выбранный, **изменить** существующий.
4. Выбрать **Final DNS** из списка тегов серверов.
5. Выбрать **Default domain resolver** (`route.default_domain_resolver`) из того же списка тегов (или «не задано»).
6. Редактировать **Rules** в текстовом поле в описанном формате.
7. При переключении на Preview / сохранении конфиг отражает изменения DNS и `default_domain_resolver`.

## Состояние и персистентность

### Общий контекст

Файл `bin/wizard_states/state.json` (и именованные снимки в той же папке) сериализует `WizardStateFile` (`ui/wizard/models/wizard_state_file.go`): `version`, `parser_config`, `config_params`, `selectable_rule_states`, `custom_rules`, метаданные. Секция DNS должна восстанавливаться при открытии визарда и при **Save current state** / **Save state as** так же, как правила и `route.final`.

### Как сохранить DNS в `state.json` (предлагаемая схема)

**1. `route.default_domain_resolver` — через существующий массив `config_params`**

По аналогии с `route.final` (`extractConfigParams` / `restoreConfigParams` в `presenter_state.go`):

- Запись: объект `{ "name": "route.default_domain_resolver", "value": "<tag>" }`, где `<tag>` — выбранный тег из `dns.servers`.
- Режим «не задано»: **не добавлять** такой параметр в `config_params` (или не сохранять пустое значение — в PLAN выбрать одно поведение и применить при восстановлении: сбрасывать поле в модели и при сборке конфига опускать ключ в `route`).

Так сохраняется единый стиль с уже существующими override-полями маршрута.

**2. Остальное содержимое вкладки DNS — ключ верхнего уровня `dns`**

Чтобы не упираться в строковые пары `config_params` для массивов объектов и многострочного текста правил, в JSON состояния добавляется объект **`dns`** (корень `state.json`, не путать с `config.dns` в итоговом `config.json`).

| Поле в объекте `dns` (state) | Тип | Назначение |
|------------------------------|-----|------------|
| `servers` | массив JSON-объектов | Как sing-box `dns.servers` в сгенерированном конфиге. |
| `rules_text` | строка | Дословное содержимое многострочного редактора правил (включая пустые строки и строки-комментарии `#…`). |
| `final` | строка | Тег сервера для `dns.final`. |
| `strategy` | строка, опционально | Как в шаблоне / sing-box (`ipv4_only`, …). |
| `independent_cache` | bool, опционально | Как в шаблоне. |

Прочие поля секции DNS шаблона, если появятся и будут поддержаны визардом, либо включаются в этот объект `dns` явным списком полей, либо через `json.RawMessage` / map для round-trip — уточнение в PLAN.

**3. Версия формата `state.json`**

- Добавление опционального ключа **`dns`** **совместимо** со старыми файлами: при отсутствии ключа после `LoadState` модель DNS инициализируется из эффективного шаблона.
- При необходимости явной миграции или строгой валидации — инкремент `WizardStateVersion` и ветка в `UnmarshalJSON` (как для `parser_config` / правил).

**4. Инициализация**

- Нет `state.json` или нет ключа `dns` / нет нужных `config_params`: взять `config.dns` и `route.default_domain_resolver` из **эффективного** шаблона; для `rules_text` — сериализовать текущий массив `dns.rules` в текстовый формат (по одному JSON-объекту на строку), чтобы поле редактора не было пустым.

**Пример фрагмента `state.json` (иллюстрация)**

```json
{
  "version": 2,
  "created_at": "2025-03-21T12:00:00Z",
  "updated_at": "2025-03-21T12:00:00Z",
  "parser_config": { ... },
  "config_params": [
    { "name": "route.final", "value": "proxy-out" },
    { "name": "route.default_domain_resolver", "value": "direct_dns_resolver" }
  ],
  "dns": {
    "servers": [
      { "type": "local", "tag": "local_dns_resolver" },
      { "type": "udp", "tag": "direct_dns_resolver", "server": "8.8.8.8", "server_port": 53 }
    ],
    "rules_text": "{\"rule_set\":\"ru-domains\",\"server\":\"yandex_doh\"}\n{\"server\":\"google_doh\"}\n",
    "final": "direct_dns_resolver",
    "strategy": "ipv4_only",
    "independent_cache": false
  },
  "selectable_rule_states": [ ... ],
  "custom_rules": [ ... ]
}
```

Реализация: расширить `WizardStateFile` полем с тегом сериализации **`json:"dns"`**, кастомные `MarshalJSON` / `UnmarshalJSON` при необходимости, `CreateStateFromModel`, `LoadState` / восстановление модели и синхронизацию с вкладкой DNS.

## Критерии приёмки

1. Вкладка DNS видна в визарде; на ней отображаются все серверы из начального `config.dns.servers` шаблона.
2. Пользователь может добавить и удалить DNS-сервер; теги уникальны; некорректные комбинации обрабатываются с сообщением об ошибке.
3. Поле **Final** — выбор строго из текущих тегов `dns.servers` (плюс согласованное поведение при удалении текущего final).
4. Поле **Default domain resolver** — выбор тега из `dns.servers` или отсутствие значения; результат в `route.default_domain_resolver`.
5. Текстовое поле **Rules** задаёт массив `dns.rules`; валидный шаблонный набор правил отображается и сохраняется без потери смысла.
6. Сгенерированный preview и сохранённый `config.json` содержат согласованные секции `dns` и обновлённый `route.default_domain_resolver` при его задании во визарде.
7. Соблюдены **SPECS/CONSTITUTION.md**: логирование, сообщения пользователю, язык UI, отсутствие лишнего шума в логах.

## Вне скоупа (явно)

- Редактирование `route.rules` или `route.rule_set` с этой вкладки (только `default_domain_resolver` в `route`).
- DNS-серверы, не описанные в sing-box / не поддерживаемые движком приложения — не требуется специальная поддержка сверх разумной валидации JSON.
- Автоматический подбор тегов outbounds для `detour` — достаточно сохранять поле как в JSON (пользователь/шаблон задают тег).

## Зависимости и документация

- Шаблон: `bin/wizard_template.json` (секция `config.dns`, пример `route.default_domain_resolver`).
- Сборка конфига: `ui/wizard/business/create_config.go` — потребуется ветка для `dns` и согласование с `buildRouteSection` для `default_domain_resolver`.
- После реализации: обновить `docs/release_notes/upcoming.md`, при необходимости `docs/ARCHITECTURE.md` и пользовательскую документацию по визарду.
