# Спецификация: Локальное скачивание пользовательских правил «SRS»

## 1. Проблема

### 1.1 Текущее поведение

- В рамках задачи `SPECS/014-F-C-RULE_TYPE_SRS_URL` в визарде добавлен тип пользовательского правила **SRS**, который создаёт **remote rule-set'ы** по URL (в т.ч. на GitHub / сторонних хостингах).
- В рамках задачи `SPECS/004-F-C-SRS_LOCAL_DOWNLOAD` реализовано **локальное скачивание SRS** и кнопка `⬇/🔄/✔️` для **встроенных** правил из шаблона (`selectable_rules`), с сохранением файлов в `{ExecDir}/bin/rule-sets/{tag}.srs` и использованием `type: "local"` в сгенерированном конфиге.
- Для **пользовательских SRS-правил** (тип «SRS» из 014) сейчас:
  - SRS всегда остаются **remote** — sing-box загружает их при старте по URL;
  - нет кнопки скачивания на вкладке Rules;
  - нет локального кэша в `bin/rule-sets` и возможности передать готовый набор SRS другому пользователю вместе с конфигом.

### 1.2 Последствия

- Пользовательские SRS-правила работают не так, как встроенные правила с SRS:
  - при каждом запуске требуется доступ к удалённым URL;
  - первый запуск/работа невозможны при блокировках доменов (аналогично исходной проблеме с `raw.githubusercontent.com`);
  - пользователь не может подготовить «офлайн-набор» (конфиг + `bin/rule-sets`) для других машин.
- UX на вкладке Rules становится несогласованным: встроенные правила с SRS имеют явный индикатор состояния (скачано/не скачано), а пользовательские — нет.

---

## 2. Цели и границы

### 2.1 Цели

- **Унифицировать UX** для всех правил SRS (встроенных и пользовательских):
  - добавить возможность локального скачивания и перезагрузки SRS для пользовательских правил типа «SRS»;
  - использовать локальные файлы `bin/rule-sets/{tag}.srs` в сконфигурированном `rule_set`, когда они доступны.
- **Снизить зависимость от сети** для пользовательских SRS-правил:
  - правило может работать при отсутствии доступа к исходному URL, если SRS были скачаны ранее;
  - конфиг + `bin/rule-sets` можно переносить между машинами так же, как для встроенных правил.

### 2.2 Вне рамок

- Изменение формата wizard state и структуры `PersistedCustomRule` сверх того, что уже определено в `SPECS/014-F-C-RULE_TYPE_SRS_URL` и `SPECS/018-F-C-CUSTOM_RULE_SUBSYSTEM_REFACTOR` (источником правды оставляем уже согласованную модель: remote rule_set + rule, плюс, при необходимости, минимальные расширения для хранения локального состояния).
- Любые изменения UX диалога **Add/Edit Rule** для типа «SRS», не связанные с индикатором/кнопкой скачивания (двухуровневый каталог runetfreedom, поле ручных URL и пр. остаются как в 014).
- Автоматическое фоновое обновление пользовательских SRS по расписанию; в этой задаче — только явное действие пользователя (кнопка/авто-скачивание при первом открытии Rules).

---

## 3. Требования

### 3.1 Кнопка SRS для пользовательских правил

- На вкладке **Rules** для **каждого пользовательского правила типа «SRS»** (из wizard state) добавить такую же маленькую кнопку после иконки `?`, как для встроенных правил SRS:

| Состояние | Иконка | Описание |
|----------|--------|----------|
| SRS отсутствует | `⬇` | Локального файла `{ExecDir}/bin/rule-sets/{tag}.srs` нет **хотя бы для одного** tag правила. Клик → запуск скачивания недостающих SRS. |
| Скачивание | `🔄` | Идёт загрузка **всех недостающих** SRS. Кнопка неактивна. |
| Скачано | `✔️` | Для **всех** tag правила SRS присутствуют локально. Клик → принудительная перезагрузка (обновление) всех SRS этого правила. |

- Логика определения состояния (`⬇/🔄/✔️`) и поведения на клик должна быть **максимально переиспользована** с реализованной в задаче `004-F-C-SRS_LOCAL_DOWNLOAD` (одна кнопка на правило, множественные SRS поддерживаются).

### 3.2 Блокировка чекбокса до скачивания

- Для пользовательских правил типа «SRS» действуют те же ограничения, что и для встроенных:
  - чекбокс включения правила **заблокирован**, пока не скачаны все SRS этого правила;
  - при клике по заблокированному чекбоксу показывается попап (текст на английском, по аналогии с 004):  
    `First download SRS by clicking the ⬇ button.`
  - после успешного скачивания всех SRS чекбокс разблокируется и может быть включён.

### 3.3 Локальное хранение и формат rule_set

- Директория и имена файлов — те же, что для встроенных SRS:
  - путь: `{ExecDir}/bin/rule-sets/`;
  - имя файла: `{tag}.srs` (tag для пользовательских правил уже формируется в 014, формат `custom-<name>`).
- Для пользовательских SRS-правил в сгенерированном конфиге:
  - при наличии локального файла для tag:
    - `type: "local"`, `format: "binary"`, `path: "<локальный путь>"` (как для встроенных);
  - при отсутствии локального SRS (или если пользователь явно не скачивал):
    - поведение как сейчас в 014: `type: "remote"`, `format: "binary"`, `url: "<URL>"`.
- Логика генерации конфига должна:
  - использовать **единый механизм** подстановки `local`/`remote` для rule_set как для встроенных, так и для пользовательских SRS;
  - не ломать уже существующие конфиги/состояния, где SRS ещё не скачаны.

### 3.4 Поведение при первом открытии визарда

- Для пользовательских SRS-правил с `Enabled == true` в wizard state:
  - при первом открытии вкладки **Rules** (в рамках сессии визарда) поведение аналогично встроенным правилам с `default: true`:
    - если локальных SRS нет — автоматически запустить скачивание (кнопка `🔄`);
    - при успехе состояние `✔️`, правило остаётся включённым;
    - при ошибке — правило переводится в состояние «выключено», кнопка `⬇`, показывается диалог ошибки.
- Для правил с `Enabled == false` авто-скачивание не запускается, пока пользователь явно не нажмёт `⬇`.

### 3.5 Ошибки и диалоги

- При ошибке скачивания SRS для пользовательского правила:
  - использовать уже существующий диалог из `SPECS/005-F-C-DOWNLOAD_FAILED_MANUAL` (тот же стиль и тексты);
  - правило остаётся выключенным, кнопка возвращается в состояние `⬇`.
- Частично скачанные файлы:
  - при отмене/ошибке частичный файл не должен использоваться (логика такая же, как для встроенных SRS).

## 4. Критерии приёмки

1. На вкладке **Rules**:
   - у всех пользовательских правил типа «SRS» отображается кнопка `⬇/🔄/✔️` по тем же правилам, что и для встроенных;
   - чекбоксы таких правил заблокированы, пока SRS не скачаны.
2. Скачивание/перезагрузка SRS для пользовательских правил:
   - запускается по клику `⬇`/`✔️`;
   - корректно обрабатывает правила с несколькими SRS (все теги этого правила);
   - при успехе все нужные файлы появляются в `{ExecDir}/bin/rule-sets/{tag}.srs`.
3. В сгенерированном конфиге:
   - при наличии локальных файлов соответствующие rule_set имеют `type: "local"` и `path` (remote URL больше не используется в конфиге);
   - при отсутствии локальных файлов поведение идентично текущему (remote rule_set).
4. Авто-скачивание при первом открытии Rules:
   - для включённых пользовательских SRS-правил запускается автоматически;
   - при успехе правила остаются включёнными, при ошибке — выключаются и показывается диалог.
5. Обратная совместимость:
   - существующие wizard state с пользовательскими SRS-правилами продолжают открываться;
   - при отсутствии локальных SRS поведение полностью совпадает с текущей версией до первого скачивания.

---

## 5. Связанные документы и реализация

- Локальное скачивание SRS для встроенных правил: `SPECS/004-F-C-SRS_LOCAL_DOWNLOAD`.
- Тип пользовательского правила «SRS»: `SPECS/014-F-C-RULE_TYPE_SRS_URL`.
- Рефакторинг подсистемы пользовательских правил: `SPECS/018-F-C-CUSTOM_RULE_SUBSYSTEM_REFACTOR`.
- Обработка ошибок скачивания SRS: `SPECS/005-F-C-DOWNLOAD_FAILED_MANUAL`.
