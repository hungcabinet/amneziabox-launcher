# План реализации: WIZARD_SETTINGS_TAB (032)

Опирается на **SPEC.md** в этой папке и **SPECS/CONSTITUTION.md**, **SPECS/IMPLEMENTATION_PROMPT.md**.

## 0. Цель этапа

- Вкладка **Settings** в визарде (порядок табов — в SPEC).
- Секция **`vars`** в **`wizard_template.json`**, массив **`state.vars`** в **`state.json`**.
- Разрешение значений → фильтрация **`params`** (платформа + **`if`**) → мерж → подстановка маркеров **`@<name>`** только для объявленных имён.
- Перенос галочки TUN macOS с **Rules** на **Settings**; **`tun`** + **`"if": ["tun"]`** у TUN-**`inbounds`** с **`platforms": ["darwin"]`**.
- **`clash_secret`:** случайный дефолт **в коде** (16 символов **`[A-Za-z0-9]`**), без логирования значения (CONSTITUTION).

**Планка качества:** делать сразу **солидно** — тот же уровень аккуратности, что у соседних вкладок визарда (DNS, Rules): поведение предсказуемо, миграции и краевые случаи закрыты, без философии «сначала черновик». Список переменных и объём — **полный** (см. SPEC, **раздел 13**).

Ниже — порядок работ. Утверждённые решения по продукту — **раздел 13**. Детали формата **`text_list`** и белого списка путей для **`@…`** в **`config`** — при реализации / в **TASKS.md**.

## 1. Шаблон (`wizard_template.json`)

1. Добавить корневой массив **`vars`** (сводная таблица в SPEC): **`tun_address`**, **`tun_mtu`**, **`tun`**, **`mixed_listen_port`**, **`log_level`**, **`clash_api`**, **`clash_secret`**, при необходимости **`apply_tun_sniff_rules`** и др.
2. В **`config`**: литералы **`@clash_api`**, **`@clash_secret`**, **`@log_level`** в согласованных узлах (белый список путей — п. 4).
3. В **`params`**: заменить фиксированные значения на **`@…`** где нужно (TUN **`address`**, **`mtu`**, mixed **`listen_port`** на **`darwin`**); для macOS TUN-блока добавить **`"if": ["tun"]`**.
4. Валидация при загрузке: каждый **`@<name>`** имеет объявление в **`vars`**; каждый **`if`** ссылается на существующее **`name`**; **`name`** уникальны среди переменных; элементы **`{"separator": true}`** без **`name`** (см. **`validateVarsSeparator`** в **`template_validate.go`**).

**Файлы:** `bin/wizard_template.json`, парсинг в `ui/wizard/template/loader.go` (или вынесенный парсер структуры корня шаблона).

## 2. Структуры Go

1. **`TemplateVar`**: поля как в SPEC, включая **`Separator bool`**, **`default_value`** как **`VarDefaultValue`** (скаляр или объект платформенных ключей — **`vars_default.go`**).
2. Расширить корневую структуру десериализации шаблона: **`Vars []TemplateVar`**.
3. **`TemplateParam`:** поле **`If []string`** (имена из **`vars`**), опционально; пусто = как сейчас.
4. **`TemplateData`:** хранить **`Vars`**, при необходимости кэш map **`name` → var** для быстрого доступа.

**Файлы:** `ui/wizard/template/loader.go`, при росте — `ui/wizard/template/vars.go` (или рядом).

## 3. Разрешение переменных (`ResolveVars` / аналог)

Вход: объявления **`vars`**, **`state.vars`**, сырой корень шаблона (для **`default_node`**), опционально уже распарсенный **`config`** как `map`/`json.RawMessage`.

Для каждого **`name`** из **`vars`**:

1. Если есть запись в **`state.vars`** — распарсить **`value`** по **`type`** (строка / **`text_list`** → **`[]string`**, в файле — строки через **newline**, раздел **13** п. 6).
2. Иначе **`default_value`**, иначе чтение узла по **`default_node`** (точечная нотация от корня шаблона — **`config`**, при необходимости **`params`**, зафиксировать в коде).
3. Пустой итог: скаляр → `""`; **`text_list`** → пустой срез; дальше подстановка и warn — по SPEC.

**Особый случай — `clash_secret`:**

- Если после шагов 1–2 значение **пустое** **или** строка с префиксом **`CHANGE_THIS_`** (как в текущем шаблоне) — сгенерировать: длина **16**, алфавит **`A–Z` `a–z` `0–9`**, **`crypto/rand`**.
- В **`state.json`** сгенерированное попадает **только при обычном сохранении state** пользователем (единый цикл Save; автосохранения в проекте нет). До Save значение может жить в памяти для Preview — по необходимости реализации.

Нигде не логировать значение секрета (CONSTITUTION / **`internal/debuglog`**).

---

## 4. Платформа и `if` на `params`

1. Условие TUN на macOS — только **`params.if` / `if_or`** и bool **`tun`** (**`vars`**, **`state.vars`**); **`matchesPlatform`** не использует синтетическую платформу **`darwin-tun`** (удалена). **`config_params.enable_tun_macos`** мигрировать при загрузке state (раздел **8. Миграция**).
2. В **`params[].if`** в **первой версии движка** допускаются **только** имена переменных с **`type`: `bool`**; условие истинности — то же, что для **`tun`**: истинно только **`"true"`**.
3. Порядок: как сейчас в SPEC — сначала фильтр по платформе (**`matchesPlatform`**), затем по **`if`**.

**Файлы:** `ui/wizard/template/loader.go` (`applyParams`, `matchesPlatform`), вызовы **`GetEffectiveConfig`** / сборка конфига в `ui/wizard/business/create_config.go`, презентер.

## 5. Подстановка `@<name>`

1. Работать по **списку объявленных имён** (SPEC: не регекс по всему JSON-тексту). Для каждого **`name`**: искать строковые литералы, **равные** `"@" + name`, в разрешённых местах дерева (после мержа **`params`** в копию **`config`**).
2. **`text`**, **`bool`**, **`enum`**, **`custom`** → одна JSON-строка; **`tun_mtu`** / **`mixed_listen_port`** — из строки в число для JSON. Отдельной жёсткой валидации на Settings не требуется: при невалидном вводе — цепочка дефолтов шаблона и **warn** при сборке по SPEC (как для пустого итога у **`@…`**).
3. **`text_list`** → подстановка JSON-массива строк в узел, описанный в шаблоне (синтаксис одного маркера на весь массив vs массив из одного элемента-маркера — зафиксировать в TASKS по SPEC).
4. Оставшиеся неизвестные **`@…`** → валидация на этапе загрузки шаблона; после подстановки сырых маркеров быть не должно; пустой итог → warn + `""` / `[]` — по SPEC.

**Файлы:** новый модуль или `loader.go` / `create_config.go`, общая точка перед финальной сериализацией конфига.

## 6. UI: вкладка Settings

1. Порядок табов: **Settings** — **предпоследняя** перед **Preview** (как в SPEC): … → Rules → **Settings** → Preview (полный список табов — в **`wizard.go`** при вставке).
2. Строить строки из **`TemplateData.Vars`**: фильтр **`wizard_ui`**, **`platforms`** против текущей ОС.
3. Контролы: **`text`**, **`bool`**, **`enum`**, **`text_list`** — для **`text_list`** одно **многострочное** поле, строки списка разделять **переносами**; разумная минимальная высота ~**3** строки (как у соседних форм).
4. Режим дефолта / «Сброс» / отсутствие записи в **`state.vars`** — по SPEC.
5. Подписи: текст из **`comment`** в **`vars`** (**английский**), формулировки — нормальные пользовательские строки, не черновик; отдельные ключи **`wizard.settings.*`** в **`en.json`** не обязательны, пока в проекте не принято иное (можно добавить сразу, если выгоднее единообразию с другими табами).
6. **macOS:** снятие bool **`tun`** — блок при **`RunningState.IsRunning()`**; иначе привилегированное удаление кеша **`experimental.cache_file`** под **`bin/`** (если есть) и **`logs/sing-box.log`** / **`.old`** под **`ExecDir`** (см. SPEC, **docs/CREATE_WIZARD_TEMPLATE**, **`settings_tun_darwin.go`**).

**Файлы:** `ui/wizard/tabs/settings_tab.go`, **`settings_tun_darwin.go`** / **`settings_tun_stub.go`**, `ui/wizard/wizard.go`, `ui/wizard/presentation/*.go`, `ui/wizard/models/wizard_model.go`.

## 9. `clash_secret` (код)

1. Функция генерации: 16 байт из алфавита 62 символа, равномерно (карта из **`crypto/rand`** в индексы).
2. Точка вызова: при разрешении **`clash_secret`**, если значение пустое или с префиксом **`CHANGE_THIS_`** (раздел **13**).
3. Не выводить в **Info/Debug** логи; ошибки чтения **`crypto/rand`** — по CONSTITUTION (сообщение пользователю без утечки секрета).

**Файлы:** `internal/…` или `ui/wizard/business/` — по конвенции проекта (проверить **docs/ARCHITECTURE.md**).

## 10. Тесты

- Разрешение **`vars`** с **`default_node`** / **`default_value`** / **`state.vars`**.
- **`if`** / **`if_or`** на **`params`**: логика на bool-**`vars`**; если **`vars[].platforms`** не включает текущую ОС (**`VarAppliesOnGOOS`**), переменная для условия **ложна**, даже при **`"true"`** в **`state.vars`** (см. SPEC § условные params, **`vars_resolve.go`**, тесты в **`vars_resolve_test.go`**).
- Подстановка **`@name`**: целый литерал, порядок имён при **`tun`** vs **`tun_address`**.
- **`text_list`**: сериализация / парсинг **`value`**.
- Миграция **`config_params.enable_tun_macos`** → **`tun`**.
- Генерация **`clash_secret`**: длина, алфавит, стабильный тест с подменой **`rand.Reader`**.

Пакеты GUI по CONSTITUTION не подключать к **`go test ./...`** там, где запрещено — юнит-тесты бизнес-логики и шаблона отдельно.

## 13. Решения (утверждено)

1. **`clash_secret`:** генерировать, если после разрешения значение **пустое** **или** начинается с префикса **`CHANGE_THIS_`** (как у плейсхолдера в шаблоне).
2. **Персистентность:** только существующий цикл сохранения **`state.json`**; автосохранений нет — отдельного сценария «записать секрет без Save» не вводить.
3. **`tun` и `params.if`:** только переменные с **`type`: `bool`**; истинность — строка **`"true"`** (остальное — ложь для фильтра **`params`**).
4. **`tun_mtu` / `mixed_listen_port`:** без отдельной жёсткой валидации в UI; опора на дефолты шаблона и **warn** + подстановка по SPEC при невалидных/пустых итогах у **`@…`**.
5. **Табы:** **Settings** — предпоследняя перед **Preview** (как в SPEC).
6. **`text_list`:** одно многострочное поле, элементы списка — **по строкам** (newline); ~**3** строки видимой высоты минимум.
7. **Подписи:** **`comment`** в **`vars`** (англ.), текст нормальный; **`en.json`** — опционально, см. **§6** п. 5.
8. **`state.json`:** ключ массива переопределений — **`"vars"`**.
9. **Объём:** **полный** набор из SPEC и сводной таблицы (**включая** **`text_list`**, **`mixed_listen_port`**, все перечисленные переменные), без урезания «на потом».

Задача **032** закрыта (**032-F-C**); чеклист — **TASKS.md** (выполнен). При смене продуктовых решений править **§13** и **SPEC.md**.
